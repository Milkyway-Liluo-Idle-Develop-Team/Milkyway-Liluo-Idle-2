# 技术说明

本项目应该采用以下的技术进行实现：

前端：独立部署的vite服务器
后端：python flask + flask websocket + sqlite + sqlalchemy

## 核心数据
核心的游戏数据和说明位于 base/ 目录下，其中action.json是最关键的内容。

## 前端
核心游戏内容采用websocket来进行通信，通过websocket来进行游戏相关的请求以及接受更新内容。

要求充分的考虑工程化思想，做好模块拆分和通用功能的复用。

### 前端风格
类似银河奶牛放置

### 前端细节

前端也要获取到action.json，并通过里面的一些信息在前端执行相关的模拟和判断。

前端需要在需要的时候去主动对后端请求更新和同步，而考虑到延迟和误差，前端需要在时序上做一些调整。


比如说在用户的视角里，当前的活动进度条满了后，就应该结算并看到物品栏里物品增加了。
这个过程我们需要在时序上做一些调整。首先我们在0s的时候，设置了活动队列（或者说从服务端获取到当前的记录时间位置），然后客户端展示进度条的时候不要从0s开始，而是略微延迟一点展示（也就是展示的进度条略微落后服务端的进度，比如0.2s），然后游戏逻辑预期在1s的时候产出1个木头，但是我们前端实际上还要延迟0.2s到1.2s才填满进度条，此时前端向后端发出同步请求，然后后端实际执行后返回结果（前端不需要在这里进行等待，不然会让玩家感受到网络和计算延迟，前端进度条直接进入下一步即可）。通过这样的处理来使得玩家不会感受到后端不定期更新、网络和计算延迟带来的不适感。

特殊情况是执行活动的速度过快的情况（比如执行一次活动的耗时已经小于1s甚至更小（比如0.1s）），这种情况下我们不应该按照上面的逻辑进行，否则会导致高频的请求。这种情况下我们前端不展示从0到1的进度条，转而展示一个持续滚动的进度条的同时展示 x次行动/s 的信息，然后按照1s一次的频率来执行更新。


## 后端

核心游戏内容通过websocket来和前端通信，只在接受到前端的核心请求后才更新游戏进度，然后通过推送数据增量更新的数据包来通知前端更新，我们不维护一个持久的线程来模拟游戏进程。

要求充分的考虑工程化思想，做好模块拆分和通用功能的复用。
充分利用ORM框架的最佳实践来进行设计。

管理好核心的生命周期，善于利用生命周期的上下文系统来优化系统设计：
1. http请求的生命周期（通过flask的g）
2. websocket内消息处理的生命周期
3. 游戏逻辑相关的生命周期

### 后端核心系统细节

我们是一个idle游戏，核心的逻辑就是设置活动队列、推进一段时间的进度。
另外还有一个战斗系统，但是这个战斗系统和活动队列系统进行分离。

我们只在前端进行了特定的同步请求的时候推进进度到当前时间（这里主要指的是战斗系统以外的行为）。

对于战斗系统，在战斗开始的时候锁定相关的核心数据不许更改，并创建数据快照到战斗系统内。
我们以2s为固定间隔来生成这段时间内的战斗内容的播片（在服务端生成播片的时候已经实际进行了相关结算），客户端做的是定期请求并获取到最近的播片片段并在客户端进行播放（注意后端是在客户端做请求后才实际的计算并生成播片所以客户端实际需要稍微提前一点进行请求）


### 后端游戏系统细节
需要有RPG游戏类似的属性系统（有属性修饰器系统），物品系统

物品系统需要着重思考和设计，需要满足以下的条件：
1. 我们可能需要在不同的地方用到不同的物品系统（比如玩家背包、战斗临时背包等的），所以物品系统需要尽可能的可复用。
2. 我们需要在一些生命周期内追踪物品的更改状况，也就是说我们的物品系统需要有diff系统
3. 物品的定义应该是(id, item_state)类的复合定义，item_state是一个32位数字，用来描述一个物品的变种（具体规则交由物品注册的时候进行定义），在默认语义下默认item_state为0，且可缺省描述（缺省的时候即默认为0）
4. 物品的数量支持小数点部分的累计（比如一个活动奖励1木头，但是在等级或者装备加持下有20%的产量提升，此时的活动奖励记录为1.2木头，此时的小数部分（0.2）应该被累计起来，累计超过1的时候就视为1个真实数量添加上去）
