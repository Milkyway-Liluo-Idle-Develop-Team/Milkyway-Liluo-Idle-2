# 技术说明

本项目应该采用以下的技术进行实现：

前端：独立部署的 Vite 服务器
后端：Go + sqlc

## 核心数据

核心的游戏数据和说明位于 `base/` 目录下，结构如下：

| 文件/目录 | 说明 |
|-----------|------|
| `actions.json` | 游戏核心数据配置，定义所有物品（items）和事件（events）的元数据 |
| `actions_format.md` | `actions.json` 的格式说明文档 |
| `attributes.json` | 属性定义及静态修饰器模板，与 `actions.json` 同等对待 |
| `游戏文档/` | 各系统的详细设计文档 |

`actions.json` 和 `attributes.json` 均通过 `//go:embed` 嵌入二进制，启动时加载并校验。变更后需运行 `go run ./cmd/genregistry` 更新数字 ID 映射。

## 前端

核心游戏内容采用 WebSocket 进行通信，通过 WebSocket 进行游戏相关的请求以及接受服务端主动推送的增量更新。

要求充分考虑工程化思想，做好模块拆分和通用功能的复用。

### 前端风格

类似银河奶牛放置（Milkyway Idle），采用三栏式布局：左侧属性面板 + 中间场景面板 + 右侧库存面板，顶部固定标题栏。信息密度优先，操作路径最短。

详细 UI 设计见 `base/游戏文档/UI/` 目录下的五个文档：
- `0-UI设计概览.md` — 整体布局、设计原则、状态同步机制
- `1-顶部栏.md` — 品牌区、当前执行状态、队列快捷下拉、功能按钮
- `2-左侧属性面板.md` — 生产属性、升级行动、战斗属性的可折叠分组展示
- `3-中间场景面板.md` — 场景切换、事件卡片、事件可见性
- `4-右侧库存面板.md` — 物品/装备标签切换、分类卡片式布局
- `5-交互与状态同步.md` — 增量更新处理优先级、断线重连、操作反馈

### 前端同步机制

前端也需要获取到 `actions.json`，用于：
- 展示事件列表、物品信息、需求条件等
- 在前端执行可见性判断（requirements 内的 event/item/fluid 已在图鉴内解锁才展示）
- 展示物品和事件的名称、描述等本地化内容

前端**不主动触发结算**，而是依赖服务端的主动推送：

- **完整状态**：连接建立时接收一次完整快照，初始化所有面板。
- **增量更新**：服务端推进过程中有变更时主动推送增量包，前端按优先级局部刷新（库存变化 → 事件执行 → 队列更新 → 技能经验 → 属性变化 → 图鉴解锁）。
- **心跳**：客户端每 5~10 秒发送 `heartbeat`，服务端仅回复服务器时间戳 ack，用于保活和校时。
- **断线重连**：自动重连后接收完整快照，平滑恢复。

### 前端进度条处理

对于高频活动（执行耗时远小于 1s），前端不展示从 0 到 1 的单次进度条，转而展示持续滚动的进度条，同时展示 "x次行动/s" 的速率信息，按 1s 一次的频率更新。

## 后端

### 通信模型

核心游戏内容通过 WebSocket 与前端通信。

**服务端采用智能预估推送模式**：服务器为每个在线玩家维护独立的游戏推进循环，根据当前队列中事件的 `loop_time` 和进度，**智能预估下一次会产生实质性变更的时间点**，在该时间点唤醒并推进结算。有实质性变更时主动推送增量更新给客户端。客户端心跳仅用于保活和校时，**不作为触发结算的信源**。

### WebSocket 生命周期

详细的 WebSocket 生命周期管理见 `base/游戏文档/1.1-游戏运行和推进模式.md`，核心流程：

```
连接建立 → 加载玩家数据 → 构建 PlayerSession → 推送完整状态 → 启动推进循环
   → 在线期间（心跳保活 + 主动推进 + 增量推送）
   → 断开（释放资源，记录时间戳）
   → 重连（离线结算 → 推送完整状态 → 重新启动推进循环）
```

### 生命周期管理

要求充分利用生命周期的上下文系统来优化系统设计：

1. **HTTP 请求的生命周期** — 短生命周期，用于登录认证等非游戏逻辑
2. **WebSocket 连接生命周期** — PlayerSession 的创建和销毁，属性实例随连接创建/释放
3. **游戏逻辑相关的生命周期** — 事件队列推进、战斗系统、Buff 倒计时等

属性实例随 WebSocket 连接生命周期创建和销毁，不维护全局在线池。连接建立时从数据库读取玩家状态构建属性实例，连接断开后由 GC 回收。

### 推进机制

每个在线玩家的推进逻辑运行在独立的推进协程中，与 WebSocket 连接管理分离。服务端不会按固定间隔轮询，而是**根据当前队列状态智能预估下一次推送时机**：

```
while 连接存活:
    next_update = 预估当前队列下一次产出或状态变更的时间点
    sleep_until(next_update)

    elapsed = 当前时间 - 上次推进时间
    推进事件队列(elapsed)
    结算产出并应用到玩家数据
    处理更新记录 → 同类合并 → 命名空间聚合 → 构建差量包 → 推送增量更新
    上次推进时间 = 当前时间
```

**预估逻辑**：根据队列头部事件的 `loop_time`、当前 `progress` 以及资源余量，计算出下一次循环完成（或队列切换）的时间点。若队列为空或无活跃事件，则进入长睡眠等待客户端操作唤醒。

这样的好处是：
- 避免无效轮询，减少 CPU 空转
- 推送时机与游戏事件自然对齐，客户端获得更精准的进度同步
- 队列静默期（无活跃事件）不产生任何服务端开销

**离线处理**：玩家断开后推进循环停止。重连时计算离线时长，一次性执行离线结算（等同于将离线期间作为大的 delta 推进事件队列）。

### 核心系统

#### 事件系统

事件是玩家与游戏交互的核心单元，详细设计见 `base/游戏文档/2.4-事件系统.md`。

三种事件类型：

| 类型 | 说明 |
|------|------|
| `loop` | 循环事件，玩家放入队列后按 `loop_time` 周期性自动执行，是最常见的生产类事件 |
| `instant` | 即时事件，执行一次立即生效 |
| `upgrade` | 升级/解锁事件，执行后永久解锁该事件 ID |

事件队列为串行队列，支持多队列并行（通过 `queue_id` 区分）。队列采用**乐观批量结算**：一次性结算尽可能多的循环次数，直到时间耗尽、库存不足或达到目标次数。

结算规则（消耗时间模型）：队列本身不计算时间差，仅负责消耗外部传入的 `delta` 并产出内容。

执行失败与队列重排：若当前事件条件不满足，向后查找第一个满足条件的事件互换位置。

#### 属性系统

需要类似 RPG 游戏的属性系统，含有属性修饰器（Modifier）系统。详细设计见 `base/游戏文档/2.2-属性系统.md`。

核心架构分为三层：

1. **属性定义层** — 定义属性的 id、默认值、min/max、方向（positive/negative）、分组
2. **属性注册表** — 全局只读注册表，启动时从 `attributes.json` 加载，维护属性依赖图和反向依赖图
3. **修饰器系统** — 描述"如何改变属性值"的最小单元

修饰器结构：

| 字段 | 说明 |
|------|------|
| `op` | 操作类型：`ADD`（加法）→ `MULTIPLY`（乘法，效果为 ×(1+value)）→ `OVERRIDE`（覆盖），按此顺序分阶段计算 |
| `value` | 固定数值（与 `ref_attr` 二选一） |
| `ref_attr` | 引用其他属性的当前值（与 `value` 二选一），支持递归求值 |
| `display` | 前端展示方式：`FIXED` / `PERCENT` / `PER_SECOND`，仅影响展示不参与计算 |
| `source` | 来源标识（equipment / tool / skill / buff / base） |
| `priority` | 计算优先级 |

修饰器生命周期管理：通过 `AddModifiers` / `RemoveModifiers` / `UpdateModifiers` 接口按 `source` 分组维护，变更后自动标记脏缓存。

属性计算采用**懒计算 + 脏标记**机制：只在被读取且 dirty 时才重算，通过反向依赖图实现脏标记传播。支持临时计算上下文（Context）用于一次性模拟场景（事件结算、战斗模拟、预览强化结果）。

#### 物品系统

物品系统需要着重思考和设计，详细设计见 `base/游戏文档/2.3-物品系统.md`。

核心要求：

1. **可复用** — 物品系统需要在不同场景使用（玩家背包、战斗临时背包等），采用 `(ItemID, ItemState)` 复合键
2. **Diff 系统** — 物品系统需要追踪物品的更改状况，通过数据更新记录机制实现
3. **ItemState** — 物品由 `(id, item_state)` 复合定义，`item_state` 为 32 位整数，描述物品变种（强化等级、锻造词缀等），默认值为 0 且可缺省
4. **小数累计** — 数量支持小数部分累计（如 1.2 木头的 0.2 部分累计，超过 1 时转为真实数量）

物品分类（classification）：`resources`、`equipment`、`fuel`、`ores`、`important`、`animal_materials`、`fluid`

物品可能带有子详情：
- `tool_details` — 工具类型、生效位置、产出加成系数
- `equipment_details` — 装备类型、元素属性、战斗技能、穿戴位置、属性数值
- `upgrade_details` — 最大升级等级、强化/锻造槽位、升级曲线（推荐等级、成功率、属性倍率）

流体（`fluid`）本质上是物品的特殊形式，使用相同的 ItemID 和库存系统存储。

#### 经验系统

详细设计见 `base/游戏文档/2.1-经验系统.md`。

玩家拥有多个技能（伐木、采矿、制作、强化、锻造、魔法、力量等），每个技能有独立的等级和经验值。经验通过执行事件获得（事件的 rewards 中含 `type: "experience"` 的奖励），技能升级可解锁新事件、提升效率和满足装备前置条件。

#### 图鉴系统

详细设计见 `base/游戏文档/2.6-图鉴系统.md`。

记录玩家解锁的事件、物品、区域：
- **事件图鉴** — 首次触发事件时解锁
- **物品图鉴** — 首次获得物品时解锁
- **区域图鉴** — 首次在区域内执行行动时解锁

图鉴影响前端的可见性判断（未解锁的事件不展示）。

#### 区域系统

详细设计见 `base/游戏文档/2.5-区域系统.md`。

区域用于对事件进行分类和组织（对应 `actions.json` 中的 `map` 字段），通过 upgrade 事件逐步解锁新区域。当前所有事件归属 `village` 区域。

### 战斗系统

战斗系统与事件队列系统分离。在战斗开始的时候锁定相关的核心数据不许更改，并创建数据快照到战斗系统内。

### 数据更新记录机制

详细设计见 `base/游戏文档/3.1-数据更新记录机制.md`。

程序运行过程中记录各种行为造成的数据变动，是构建推送给前端的数据更新包的重要依据。

**上下文命名空间**：通过 `PushNamespace` / `PopNamespace` 管理嵌套的命名空间（基于栈结构），记录变更发生在什么过程内。例如：

```
PushNamespace("event_execution")
    // 事件系统写入 EventExecutionRecord
    // 物品系统写入 InventoryChangeRecord
    // 经验系统写入 SkillXpRecord
    // 属性系统写入 AttributeDirtyRecord
    // 图鉴系统写入 BestiaryUnlockRecord
PopNamespace()
```

**更新记录处理**：
1. 命名空间级别处理 — 同一命名空间内同类合并（如同 ItemID 累加、同 AttrID 去重）
2. 命名空间聚合 — 汇总所有命名空间的变更
3. 差量数据包构建 — 按各系统序列化格式组装 JSON 推送给前端

各系统独立定义自己的更新记录类型和合并规则：

| 系统 | 记录类型 | 合并规则 |
|------|----------|----------|
| 物品系统 | `InventoryChangeRecord` | 同 ItemID+ItemState 累加 |
| 经验系统 | `SkillXpRecord` | 同 SkillID 累加 |
| 属性系统 | `AttributeDirtyRecord` | 同 AttrID 去重 |
| 事件系统 | `EventExecutionRecord` | 同 EventID+QueueID 累加 |
| 事件系统 | `QueueUpdateRecord` | Scope 取最高优先级 |
| 图鉴系统 | `BestiaryUnlockRecord` | 同 Type+ID 去重 |

### 属性修饰器与 actions.json 的数据绑定

`actions.json` 中的装备/工具数据（如 `equipment_basic_data`、`equipment_upgrade_data`、`tool_basic_data`）在解析时自动转换为修饰器。属性字符串 ID 必须先查 `attr_registry.json` 转换为数字 `AttributeID`。如果 `actions.json` 中出现未注册的属性 ID，启动校验时报错。

### 技术细节

- 物品、事件、属性、区域均使用稳定的数字 ID（`int64`）存储，字符串 ID 仅用于前端展示和配置编辑
- 后端启动时加载并校验 `actions.json` 和 `attributes.json`，所有数据常驻内存，查询为 O(1)
- 配置变更后需运行 `go run ./cmd/genregistry` 重新生成数字 ID 映射
- 属性系统不直接依赖数据库，数据库只存"状态事实"（穿了什么、等级多少），属性系统负责翻译成修饰器并计算最终值
