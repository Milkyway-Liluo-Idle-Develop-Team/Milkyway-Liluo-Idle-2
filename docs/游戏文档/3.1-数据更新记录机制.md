# 数据更新记录机制

数据更新记录机制是后端构建推送给前端的 `state.diff` 和 `state.full` 的核心基础设施。它不是持久化日志，而是**单次游戏逻辑周期内的变更收集器**。

## 核心抽象

整个机制基于三个接口/类型：

```go
// record.go
type RecordBucket interface {
    SystemName() string
    MergeInPlace(other RecordBucket)
    SerializeDiff() (proto.Message, error)
    IsEmpty() bool
}

type SystemProvider interface {
    SystemName() string
    NewBucket() RecordBucket
    SerializeFull(state any) (proto.Message, error)
}
```

| 组件 | 职责 | 生命周期 |
|------|------|----------|
| `Recorder` | 命名空间栈，收集一个周期内的所有变更 | 单次 tick / 命令处理 |
| `RecordBucket` | 单个子系统的变更容器（去重/合并逻辑） | 随 Recorder 创建和销毁 |
| `SystemProvider` | 将完整状态序列化为 `state.full` | 全局单例 |
| `Registry` | 管理所有 Provider，负责 BuildDiff / BuildFullSnapshot | 全局单例 |

## Recorder：命名空间栈

`Recorder` 是一个短周期的变更收集器，基于栈结构支持嵌套命名空间：

```go
type Recorder struct {
    reg       *Registry
    stack     []*namespace      // 当前活跃的命名空间栈
    completed []*namespace      // 已 Pop 的命名空间，等待 BuildDiff
}
```

### 使用方式

```go
rec := mgr.NewRecorder()
rec.PushNamespace("cmd")           // 命令处理命名空间
s.SetRecorder(rec)                  // 将 recorder 附加到各子系统

// 在此阶段，各子系统的写操作会自动向 rec 的当前命名空间写入记录
s.inv.Add(item, qty)                // inventory 向 rec 写入 InventoryChangeRecord
s.best.UnlockEvent(id)              // bestiary 向 rec 写入 BestiaryUnlockRecord

s.ClearRecorder()                   // 解除 recorder 附加
rec.PopNamespace()                  // 关闭当前命名空间

// BuildDiff 聚合所有命名空间
diff, _ := mgr.Registry().BuildDiff(rec)
```

### 嵌套命名空间

允许在一个已设置命名空间的过程中再次 `PushNamespace`，产生嵌套：

```go
rec.PushNamespace("action_queue")   // 外层：事件结算
    // settlement 过程中...
    rec.PushNamespace("battle")     // 内层：战斗结算
    rec.PopNamespace()
rec.PopNamespace()
```

每个命名空间独立存储各子系统的 `RecordBucket`。`BuildDiff` 时，同名子系统的 Bucket 会跨命名空间合并。

## Registry：合并与序列化

### BuildDiff（增量更新）

```go
func (r *Registry) BuildDiff(rec *Recorder) (*pb.StateDiff, error) {
    namespaces := rec.PopAll()   // 取出所有命名空间（含已 Pop 和栈内剩余的）
    merged := make(map[string]RecordBucket)

    for _, ns := range namespaces {
        for sysName, b := range ns.buckets {
            if existing, ok := merged[sysName]; ok {
                existing.MergeInPlace(b)   // 同系统跨命名空间合并
            } else {
                merged[sysName] = b
            }
        }
    }
    return r.buildDiffProto(merged)
}
```

合并后，每个 `SystemProvider` 的 `SerializeDiff` 被调用，按注册顺序组装成 `pb.StateDiff`：

```protobuf
message StateDiff {
    repeated InventoryDiff   inventory      = 1;
    repeated AttributeDiff   attribute      = 2;
    repeated SkillXPDiff     skill_xp       = 3;
    repeated BestiaryDiff    bestiary       = 4;
    repeated EventExecutionDiff event_execution = 5;
    repeated EventQueueDiff  event_queue    = 6;
    repeated EquipmentDiff   equipment      = 7;
}
```

### BuildFullSnapshot（完整快照）

```go
func (r *Registry) BuildFullSnapshot(states map[string]any) (*pb.StateFull, error) {
    for _, name := range r.order {
        prov := r.providers[name]
        msg, _ := prov.SerializeFull(states[name])
        // 按注册顺序组装 pb.StateFull 各字段
    }
}
```

`states` 是一个 `map[string]any`，key 为系统名，`value` 为各子系统的状态对象（如 `*bestiary.State`、`*inventory.State`）。

## 各子系统的 Bucket 实现

### 物品系统 (`inventory`)

```go
type Bucket struct {
    changes map[item.Item]*changeEntry   // key: item.Item{ID, State}
}
```

- **合并规则**：同一 `item.Item`（`ID` + `State`）累加 `quantity_delta`；净数量为 0 时删除条目。
- **序列化**：`pb.StateDiff{Inventory: []*pb.InventoryDiff{...}}`

### 经验系统 (`skill_xp`)

```go
type Bucket struct {
    changes map[gameconfig.SkillID]*skillRec
}
```

- **合并规则**：同一 `SkillID` 累加 `xp_delta`，`new_level` 取最终值。
- **序列化**：`pb.StateDiff{SkillXp: []*pb.SkillXPDiff{...}}`

### 属性系统 (`attribute`)

```go
type Bucket struct {
    dirty map[attribute.AttributeID]bool
}
```

- **合并规则**：同一 `AttributeID` 去重，仅保留一条脏标记。
- **序列化**：`SerializeDiff` 时**现场计算最终值**，并收集该属性上的完整修饰器列表一并推送。
- **原因**：属性系统是懒计算的，只记录"哪些属性可能需要重新计算"，值在 diff 组装阶段才计算。

### 图鉴系统 (`bestiary`)

```go
type Bucket struct {
    entries map[string]unlockEntry   // key = type + "/" + id
}
```

- **合并规则**：同一 `type + id` 去重。
- **序列化**：`pb.StateDiff{Bestiary: []*pb.BestiaryDiff{...}}`

### 事件系统 — 执行记录 (`event_execution`)

```go
type execBucket struct {
    counts map[eventKey]int   // EventID+QueueID -> 完成循环次数
}
```

- **合并规则**：同一 `EventID` + `QueueID` 累加 `cycles`。

### 事件系统 — 队列记录 (`event_queue`)

```go
type queueBucket struct {
    st    *event.State
    marks map[int]bool   // queueID -> full=true/false
}
```

- **合并规则**：同一 `QueueID`，只要存在 `full=true`，结果为 `"full"`；否则 `"current"`。
- **特殊之处**：`SerializeDiff` 时直接读取 `event.State` 的实时队列数据构建快照，不依赖 Bucket 内存储的条目内容。

### 装备系统 (`equipment`)

```go
type Bucket struct {
    actions map[string]*pb.EquipmentDiff   // key = slot
}
```

- **合并规则**：同一 `slot` 保留最后一次操作（equip/unequip）。
- **序列化**：`pb.StateDiff{Equipment: []*pb.EquipmentDiff{...}}`

## 完整数据流示例

一次典型的 `loop` 事件结算周期：

```
Tick 触发
    │
    ▼
runTick(session, delta)
    │
    ├── Phase 1: drainCommandsLocked()
    │   │   （若有命令，如 queue_append）
    │   └── 命令执行 → 各子系统向 recorder("cmd") 写入记录
    │       inventory: +oak_logs (Add)
    │       event_queue: "full" (queue changed)
    │   └── BuildDiff → 推送 state.diff (cmd 结果)
    │
    ├── Phase 2: settlement
    │   └── ev.Settle(session, delta)
    │       └── 事件结算过程中
    │           inventory: -heat, +oak_logs (Settle 消耗/产出)
    │           skill_xp: +felling XP
    │           bestiary: unlock oak_logs (首次获得)
    │           event_execution: +3 cycles
    │           event_queue: "current" (progress 更新)
    │       └── 各子系统向 recorder("action_queue") 写入记录
    │   └── BuildDiff → 推送 state.diff (settlement 结果)
    │
    └── return dirty=true
        │
        ▼
BatchFlush (tickRound 结束)
    └── 单事务刷入所有 dirty 子系统
```

## 关键设计原则

1. **命令与结算串行化**：所有游戏状态修改都在 `runTick` 内、持有 `session.mu` 的情况下执行，命令排空和事件结算不会交错。
2. **Diff 即时推送**：`state.diff` 在 Phase 1 和 Phase 2 结束后立即推送，不等待 Tick Round 结束。
3. **进度更新抑制**：纯 `"current"` scope 的队列进度更新不推送，前端本地插值。
4. **命名空间隔离**：不同过程（cmd vs action_queue）的变更通过命名空间隔离，最终合并时跨命名空间累加/去重。
