# 事件系统

事件（Event）是玩家与游戏交互的核心单元，表示玩家可以执行的各类行动和剧情节点。

## Event 数据格式

`actions.json` 中的每个 `Event` 对象定义如下：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 唯一标识符 |
| `type` | string | 是 | 事件类型：`"loop"` / `"instant"` / `"upgrade"` |
| `name` | string | 是 | 显示名称 |
| `description` | string | 是 | 描述文本 |
| `need_skill` | string | 是 | 关联技能 ID，`"none"` 表示无 |
| `requirements` | array | 是 | 执行前提条件 |
| `map` | string | 是 | 所属地图/场景 |
| `loop_time` | number | 否 | 循环周期（秒），`type == "loop"` 时必填 |
| `rewards` | array | 否 | 完成后的产出奖励 |

> 注：`experience` 字段**不在** Event 顶层定义中。经验奖励通过 `rewards` 数组中 `type: "experience"` 的条目发放。

## 事件分类

### Loop 事件（循环事件）

最常见的生产类事件。玩家将其放入队列后，按 `loop_time` 周期性自动执行。

- 每次循环结算时：扣除消耗 → 发放奖励 → 累加经验。
- `loop_time` 支持小数（如 1.5 秒），未凑满一次循环的进度会保留在 `progress` 字段中。
- 同一时刻只有一个队列在运行（`queue_id = 0`）。

示例：`felling_oak_tree`（砍伐橡树），`loop_time: 2`，每 2 秒产出 1 个 `oak_logs` 和 20 点 `felling` 经验。

### Instant 事件（即时事件）

执行一次立即生效，不需要在队列中循环。通常用于一次性交互或特殊操作。

### Upgrade 事件（升级/解锁事件）

执行一次后**永久解锁**该事件 ID，将其加入玩家的 `unlocked_events` 集合。解锁后：

- 可能开放新的 loop 事件供玩家使用。
- 可能解锁新的物品配方。
- 推进主线或支线剧情。

## 需求 (Requirements)

`requirements` 数组描述执行事件的前置条件。每个 Requirement：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | `"skill"` / `"item"` / `"fluid"` / `"event"` |
| `id` | string | 对应资源的 ID |
| `comparison_types` | string | 比较运算符，可选 |
| `value` | number | 比较/消耗数值，可选 |

### 结算规则

- **消耗型**：`type` 为 `item` 或 `fluid`，且 `comparison_types` 为 `null`。执行时从库存扣除 `value` 数量。
- **门槛型**：带有 `comparison_types`（如 `bigger_or_equal`）或不属于消耗型。仅做条件检查，不扣除资源。
- **事件依赖**：`type: "event"` 要求玩家已完成（解锁）指定事件。

后端在启动时通过 `gameconfig.Load()` 将所有 Requirement 中的字符串 ID 预解析为数字 ID（`ResolvedItem`、`ResolvedID`等），运行时不再做字符串查找。

## 奖励 (Rewards)

`rewards` 数组描述事件完成后的产出。

### 物品奖励

```json
{
  "id": "oak_logs",
  "num": 1
}
```

- `id`：物品 ID。
- `num`：产出数量（优先读取）。
- `value`：兼容字段，当 `num` 未设置时回退读取。

### 经验奖励

```json
{
  "type": "experience",
  "value": 20.0,
  "skill_id": "felling"
}
```

- `type`：固定为 `"experience"`。
- `value`：经验数值（支持小数）。
- `skill_id`：经验归属的技能。

## 事件队列

### 队列结构

当前后端实现支持**多队列**（`queues map[int]*Queue`），但前端当前只使用 `queue_id = 0`。队列内按 `position` 升序排列，头部事件（`position = 0`）是当前正在执行的事件。

队列数据存储在 `player_active_events` 表中：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | INTEGER | 玩家 ID |
| `queue_id` | INTEGER | 队列编号，当前固定为 `0` |
| `event_id` | INTEGER | 事件数字 ID |
| `position` | INTEGER | 在队列中的顺序，从 `0` 开始递增 |
| `target_cycles` | INTEGER | 目标执行次数，`-1` 表示无限循环（默认），正整数表示执行指定次数后自动移除 |
| `progress` | REAL | 当前事件已积累的进度秒数 |
| `updated_at` | DATETIME | 最后更新时间 |

主键为 `(user_id, queue_id, position)`。

### 队列操作

客户端通过 WebSocket 发送命令：

- **`queue_append`**：将事件追加到队列尾部。
- **`queue_remove`**：移除指定位置的事件。
- **`queue_swap`**：交换两个位置的事件。
- **`queue_bring_to_front`**：将指定位置事件移到队首。
- **`execute_instant`**：立即执行一个 instant/upgrade 事件。

所有队列操作都通过 `SubmitCommand` 提交到 Session 的命令通道，在下一个 Tick 的 Phase 1 中被执行。

### 结算规则（消耗时间模型）

队列本身**不计算时间差**，仅负责**消耗**外部传入的一段时间 `delta`，并产出相应内容。`delta` 由 Tick 循环提供（即两次 Tick 之间的时间差，约 50ms）。

结算流程（`event.State.Settle`）：

1. 对队列头部事件（`position = 0`）计算 `timeCycles = int((delta + progress) / loop_time)`。
2. 计算库存允许的最大循环次数 `maxAffordable`（基于 requirements 中消耗型物品的库存量）。
3. `actual_cycles = min(timeCycles, maxAffordable, target_cycles)`。
4. 若 `actual_cycles > 0`：
   - 一次性扣除消耗 = 单次消耗 × actual_cycles
   - 一次性发放产出 = 单次产出 × actual_cycles
   - 一次性累加经验 = 单次经验 × actual_cycles
   - 调用 `ctx.AddItem` 发放物品，这会同时触发图鉴解锁 `best.UnlockItem`
   - 调用 `ctx.AddXP` 累加经验
   - 更新 `progress = (delta + progress) - float64(actual_cycles) * loop_time`
   - `target_cycles` 递减，归零后移出队列
5. 若 `actual_cycles == 0`：仅累加 `progress += delta`（受 `loop_time` 上限截断）。
6. 若事件被移除（完成或 instant/upgrade），下一个事件自动成为队首，但**不继续消耗剩余 delta**（剩余 delta 在本次结算中丢弃）。

### 执行失败与队列重排

若当前事件的**门槛型** requirements（技能等级、事件解锁等前置条件）不满足，`Settle` 会调用 `swapSatisfied` 向后查找第一个满足条件的事件并交换位置：

```go
if !st.checkReqs(ctx, ev) {
    if !st.swapSatisfied(ctx, q) {
        return remaining
    }
    continue
}
```

**但库存不足不会触发 swap**：当消耗型物品/流体库存不够时，`actual_cycles` 被限制为 0，仅推进 `progress`，不会与后续事件交换位置。

## 更新记录

事件系统通过 `record.Recorder` 记录变更。 settlement 期间使用 `"action_queue"` 命名空间：

```go
rec.PushNamespace("action_queue")
s.ev.Settle(s, delta)   // settlement 过程中各子系统向 recorder 写入记录
rec.PopNamespace()
```

### 事件执行记录

```go
type execBucket struct {
    counts map[eventKey]int   // EventID+QueueID -> 完成循环次数
}
```

**产生时机**：每次事件结算完成时，`Settle` 内部调用 `ctx.RecordEventExecution(eventID, queueID, cycles)` 写入。

**合并规则**：同一 `EventID` + `QueueID` 在同一命名空间内累加 `Cycles`。

**序列化**：

```json
{
  "event_execution": [
    {"event_id": 7, "queue_id": 0, "cycles": 3}
  ]
}
```

### 队列更新记录

```go
type queueBucket struct {
    st    *event.State
    marks map[int]bool   // queueID -> full=true
}
```

**产生时机**：

| 场景 | Scope |
|------|-------|
| 推进循环更新当前事件 `progress` | `"current"` |
| 玩家编排队列、移除事件、调整顺序 | `"full"` |
| 事件执行后自动移除（`target_cycles` 归零） | `"full"` |

**合并规则**：同一 `QueueID` 在同一命名空间内，只要存在 `"full"`，合并结果为 `"full"`；否则为 `"current"`。

**序列化**：

| Scope | 回传内容 |
|-------|----------|
| `"current"` | 仅队首事件的状态（`event_id`, `progress`, `target_cycles`） |
| `"full"` | 整个队列的完整事件列表 |

## 技术细节

- 事件使用稳定的数字 ID（`EventID`，`int64`）存储。
- 后端启动时构建索引：`eventsBySkill`、`eventsByMap`、`loopEvents`、`upgradeEvents`，查询为 O(1)。
- 事件校验在启动时完成：检测悬空引用（requirements/rewards 中指向不存在的物品/事件）、缺失必填字段等，校验失败会阻止服务器启动。
- `actions.json` 变更后需运行 `go run ./cmd/genregistry` 重新生成数字 ID 映射。
