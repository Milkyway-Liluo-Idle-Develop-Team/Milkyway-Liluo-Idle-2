# 物品系统

物品系统涵盖物品的身份模型、定义、库存管理及装备机制。

## 物品身份模型

一个物品实例由 **`(item_id, item_state)`** 联合标识。两者必须同时存在：

- `item_id`：物品模板的数字 ID（`item.ID`，`int32`），对应 `actions.json` 中的物品定义
- `item_state`：32 位变体描述符（`item.State`，`uint8`），编码强化等级、锻造词缀等变体信息

**`item_state = 0` 是默认态，不是"缺省"。** 从逻辑上不存在"只有 item_id 没有 item_state"的物品——表述时允许省略 State=0 的写法，但代码中始终以 `item.Item{ID: id, State: state}` 的完整形式传递。

## ItemDef（物品定义）

`actions.json` 中的每个 Item 对象在运行时被解析为 `item.ItemDef`——一个只读的物品定义：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 唯一标识符，如 `"oak_logs"` |
| `name` | string | 是 | 显示名称 |
| `tool` | bool | 是 | 是否为工具 |
| `equipment` | bool | 是 | 是否为装备 |
| `upgradable` | bool | 是 | 是否可升级/强化 |
| `classification` | string | 是 | 分类标签 |
| `tool_details` | object | 否 | `tool == true` 时必填 |
| `equipment_details` | object | 否 | `equipment == true` 时必填 |
| `upgrade_details` | object | 否 | `upgradable == true` 时必填 |

`ItemDef` 上的 `Modifiers(it Item, attrReg)` 方法接受完整的 `Item` 身份参数。当前 `it.State` 暂时忽略，将来通过 `upgrade_curve` 和 `ability_multiplier` 计算修饰器值。

### 分类 (classification)

| 分类值 | 说明 |
|--------|------|
| `resources` | 原材料（木头、石头等） |
| `equipment` | 可穿戴装备 |
| `fuel` | 燃料（如煤炭） |
| `ores` | 矿石 |
| `important` | 关键道具 |
| `animal_materials` | 动物材料 |
| `fluid` | 流体（特殊物品，如热量、铁流体、石流体） |

### 流体作为特殊物品

流体（如 `heat`、`iron_fluid`、`stone_fluid`）本质上是物品的一种特殊形式，使用相同的 `item.Item` 身份和库存系统存储。它们在 `actions.json` 中与普通物品并列定义，`classification` 为 `"fluid"`。事件中的 `type: "fluid"` 需求在结算时同样扣除库存中的对应物品数量。

## Inventory 系统

玩家库存由 `inventory.State` 管理，持久化到 `player_inventory` 表：

```sql
CREATE TABLE player_inventory (
    user_id      INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    item_id      INTEGER NOT NULL,
    item_state   INTEGER NOT NULL DEFAULT 0,
    quantity     REAL    NOT NULL DEFAULT 0,
    updated_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (user_id, item_id, item_state)
);
```

### 内存状态

```go
type State struct {
    userID   int64
    slots    map[item.Item]float64   // 当前数量
    dirty    map[item.Item]bool      // 需要刷盘的槽位
    recorder *record.Recorder
}
```

### 核心操作

```go
func (s *State) Add(it item.Item, qty float64)       // 增加库存
func (s *State) Deduct(it item.Item, qty float64)    // 扣除库存
func (s *State) Has(it item.Item, qty float64) bool  // 检查是否足够
func (s *State) Get(it item.Item) float64            // 获取当前数量
func (s *State) AddEquipChange(it item.Item, qty float64, isEquip bool)  // 装备变更（带 reason）
```

所有变更操作内部调用 `add()`：

```go
func (s *State) add(it item.Item, qty float64, reason pb.InventoryChangeReason) {
    s.slots[it] += qty
    if s.slots[it] == 0 { delete(s.slots, it) }
    s.dirty[it] = true
    s.record(it, qty, reason)   // 向 recorder 写入 InventoryChangeRecord
}
```

### 产出精度

结算时产出加成可能产生非整数（如 1.3 个石头）。**`quantity` 使用 REAL 存储**，保留小数部分跨结算周期累积。玩家看到的数量为 `floor(quantity)`。

```
raw = base × actualCycles × (1 + multiplier)
quantity = current + raw    -- REAL，如实累加
display = floor(quantity)    -- 整数展示
```

理由：结算可能只推进单轮（如推送时机恰好在一次循环完成后），此时截断损失无法被批量推进抵消。REAL 存储确保精度不丢失。

### 加载与刷盘

```go
func Load(ctx context.Context, q *dbgen.Queries, userID int64) (*State, error)

func (s *State) Flush(ctx context.Context, q *dbgen.Queries) error {
    for it := range s.dirty {
        q.UpsertInventory(ctx, dbgen.UpsertInventoryParams{
            UserID: s.userID, ItemID: int64(it.ID), ItemState: int64(it.State),
            Quantity: s.slots[it],
        })
    }
    s.dirty = make(map[item.Item]bool)
}
```

## 装备机制

装备是放在特定槽位的物品。将物品放入装备槽时，从 `ItemDef` 解析修饰器并挂载到属性系统。

`PlayerSession` 提供的装备接口：

```go
func (s *PlayerSession) Equip(ctx context.Context, it item.Item, slot string) error
func (s *PlayerSession) Unequip(ctx context.Context, slot string) error
```

### Equip 流程

```
Equip(it, slot):
    1. 检查物品定义存在且是 equipment
    2. 检查库存中有该物品
    3. 若槽位已有旧装备:
         - 移除旧装备的属性修饰器
         - 旧装备返回库存 (AddEquipChange(old, +1, false))
         - best.UnlockItem(old)  // 触发图鉴
         - eq.Unequip(slot)      // 记录 UNEQUIP
    4. 从库存扣除 (AddEquipChange(it, -1, true))  // 带 EQUIP reason
    5. 解析并挂载新装备的属性修饰器
    6. eq.Equip(slot, it)       // 记录 EQUIP
```

### Unequip 流程

```
Unequip(slot):
    1. 获取槽位上的物品
    2. 移除该装备的属性修饰器
    3. 物品返回库存 (AddEquipChange(it, +1, false))
    4. best.UnlockItem(it)      // 触发图鉴
    5. eq.Unequip(slot)
```

装备槽位持久化到 `player_equipment` 表：

```sql
CREATE TABLE player_equipment (
    user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    slot       TEXT    NOT NULL,
    item_id    INTEGER NOT NULL,
    item_state INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (user_id, slot)
);
```

## 更新记录

物品系统在库存发生变动时向当前命名空间写入记录。

### 库存变化记录

```go
type Bucket struct {
    changes map[item.Item]*changeEntry   // key: item.Item{ID, State}
}

type changeEntry struct {
    qty    float64
    reason pb.InventoryChangeReason   // LOOT / CONSUME / EQUIP / UNEQUIP / OTHER
}
```

**产生时机**：事件结算发放/扣除物品、制造/合成、装备穿脱时写入。

**合并规则**：同一 `item.Item`（`ID` + `State`）在同一命名空间内累加 `quantity_delta`；净数量为 0 时删除条目。

**序列化示例**：

```json
{
  "inventory": [
    {"item_id": 3, "item_state": 0, "quantity_delta": 5, "reason": "LOOT"},
    {"item_id": 1, "item_state": 0, "quantity_delta": -5, "reason": "CONSUME"}
  ]
}
```

## 技术细节

- 后端启动时通过 `gameconfig.Load()` 加载并校验 `actions.json`，所有物品数据常驻内存，访问为 O(1)。
- 物品使用稳定的数字 ID（`item.ID`，`int32`）存储在数据库中，字符串 ID 仅用于前端展示和配置编辑。
- `actions.json` 变更后需运行 `go run ./cmd/genregistry` 重新生成数字 ID 映射。
- 游戏配置层的 `Item` JSON 结构不对外暴露；调用方使用 `item.ItemDef`（运行时定义）和 `item.Item`（实例身份）。
- `AddItem` 在 `qty > 0` 时会触发 `bestiary.UnlockItem(it)`，这是物品图鉴的主要解锁途径。
