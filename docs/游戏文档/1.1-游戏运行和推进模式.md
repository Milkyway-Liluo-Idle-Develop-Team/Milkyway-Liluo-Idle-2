# 游戏运行和推进模式

## WebSocket 生命周期

```
┌─────────────┐     握手+认证      ┌─────────────┐
│   客户端     │ ───────────────> │   服务端     │
│  (断开状态)  │                   │  (等待连接)  │
└─────────────┘                   └─────────────┘
                                        │
                                        ▼
                              ┌───────────────────┐
                              │ 1. 加载玩家数据    │
                              │    (DB查询)        │
                              │ 2. 构建PlayerSession│
                              │ 3. 挂载各子系统    │
                              │ 4. 推送完整状态    │
                              │ 5. 注册到Manager   │
                              └───────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────┐
│                      在线期间                        │
│  ┌──────────┐                                      │
│  │  服务端   │  全局Tick(固定间隔) ──> 并行推进      │
│  │          │  有变更 ──> 推送增量更新(state.diff)  │
│  └──────────┘                                      │
└─────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼                               ▼
   ┌─────────────┐                 ┌─────────────┐
   │  正常断开    │                 │  异常断开    │
   │ 页面关闭/   │                 │ 网络中断    │
   │ 主动logout  │                 │             │
   └─────────────┘                 └─────────────┘
          │                               │
          └───────────────┬───────────────┘
                          ▼
               ┌───────────────────┐
               │  DetachConn()      │
               │  StartGraceTimer() │
               │  (默认30s宽限期)   │
               └───────────────────┘
                          │
              ┌───────────┴───────────┐
              ▼                       ▼
        ┌──────────┐           ┌──────────┐
        │ 宽限期内  │           │ 宽限期到  │
        │ 重连成功  │           │ GraceExpire│
        │          │           │ FlushAll   │
        │ AttachConn│           │ Remove     │
        │ StopGrace │           │ Close      │
        └──────────┘           └──────────┘
```

### 连接建立

1. **握手与认证**：客户端发起 WebSocket 连接，HTTP upgrade 请求携带 cookie/session token，由 `auth.Middleware` 认证。
2. **查找已有 Session**：`Manager` 以 `userID` 为 key 维护单在线。若该用户已有 `PlayerSession`（处于 Grace 状态），则踢掉旧连接、附加新连接、停止 Grace Timer、推送 `state.full`。
3. **创建新 Session**：若无已有 Session，调用 `Manager.CreateSession`：
   - 懒初始化新玩家（`playerinit.InitPlayer` 写入默认技能）
   - 从 DB 加载库存 `inventory.Load`、技能 `skill.Load`、事件队列 `event.Load`、装备 `equipment.Load`
   - 从 DB 加载已解锁事件 `LoadUnlockedEvents` 和已发现物品 `LoadDiscoveredItems` 到图鉴
   - 组装 `PlayerSession`，调用 `Manager.Add` 注册
   - 推送 `state.full` 完整快照
4. **注册 Grace Expire 回调**：`sess.SetOnGraceExpire` 设置回调函数，在宽限期到期时执行 `FlushAll` + `Remove`。

### 在线期间

服务端采用**全局固定间隔 Tick**，而非为每个玩家维护独立协程：

- `Manager.TickAll` 启动一个全局 `time.Ticker`（默认间隔 50ms）。
- 每个 Tick Round 内，先通过 `parallelTick` 并行处理所有在线 Session（按 CPU 核数分 worker）。
- 每个 Session 的 `runTick` 计算 `delta = now - lastTick`，执行命令排空 + 事件结算。
- Tick Round 结束后，`BatchFlush` 将所有产生脏数据的 Session 在一个 SQLite 事务中批量刷盘。
- 增量更新 `state.diff` 在 `runTick` **内部立即推送**，不等待 Tick Round 结束。

**客户端不发送应用层心跳**。服务端也不依赖心跳来判定连接存活（WebSocket 层的 `onclose` 回调负责触发 Grace Timer）。底层 `wsx.Conn.writeLoop` 会按 `WS_PING_INTERVAL`（默认 30s）发送 WebSocket ping 帧保活，但这是传输层行为，不参与游戏逻辑。

### 连接断开

当浏览器关闭页面、导航离开或网络中断时，WebSocket 连接关闭：

1. `OnDisconnect` 被触发。若当前关闭的是该 Session 的**当前连接**（防竞态），则执行 `DetachConn()`。
2. 若 Session 已无连接 `!HasConn()`，调用 `StartGraceTimer(30s)`，状态变为 `StateGrace`。
3. Grace 期间 Session 仍留在内存中，Tick 继续推进，玩家数据继续结算。
4. Grace 到期后执行 `onGraceExpire` 回调：`FlushAll` 刷盘 → `sessMgr.Remove` 从 Manager 中移除 → `Close` 释放资源。

### 重连

若在 Grace 期间重连：
1. `OnConnect` 发现已有 Session，复用内存中的 `PlayerSession`。
2. 踢掉旧连接（若存在）、附加新连接、停止 Grace Timer、恢复 `StateActive`。
3. 发送 `state.full` 完整快照（此时快照包含 Grace 期间累积的所有变更）。

**不存在"离线结算"概念**。断开期间若仍在 Grace 内，Tick 正常推进；若 Grace 已过，Session 已销毁，下次连接从 DB 重新加载。

## 推进机制

### Tick 循环架构

```
全局 Ticker (50ms)
    │
    ▼
┌─────────────────────────────────────┐
│ tickRound                           │
│  1. getAllSessions() 快照所有Session│
│  2. parallelTick(sessions, now)     │
│     ├── Worker 0: session[0..N]     │
│     ├── Worker 1: session[N..M]     │
│     └── ... (按CPU核数分片)         │
│  3. BatchFlush(ctx, database, results)│
└─────────────────────────────────────┘
```

`parallelTick` 对每个 Session 执行：

```
delta = now.Sub(session.lastTick).Seconds()
session.elapsedAccum += delta
runTick(session, mgr, session.elapsedAccum)
session.elapsedAccum = 0
```

### runTick 双阶段执行

`runTick` 在持有 `session.mu` 的情况下执行，分为两个阶段：

**Phase 1: 排空命令（Command Drain）**

```go
rec := mgr.NewRecorder()
rec.PushNamespace("cmd")
s.SetRecorder(rec)
hasCmds := s.drainCommandsLocked()  // 处理所有挂起的WS命令
s.ClearRecorder()
rec.PopNamespace()

if hasCmds {
    cmdDiff, _ := mgr.Registry().BuildDiff(rec)
    if !isStateDiffEmpty(cmdDiff) {
        c.Send(Outbound{Type: "state.diff", Payload: cmdDiff})
    }
}
```

所有游戏状态修改（队列操作、事件执行、装备穿脱等）都通过 `SubmitCommand` 提交到 `commandCh`（容量64），在 `drainCommandsLocked` 中批量执行。这保证了**命令执行与事件结算串行化**，避免竞态。

**Phase 2: 事件结算（Settlement）**

```go
if s.ev != nil && s.ev.HasActive() {
    rec := mgr.NewRecorder()
    s.SetRecorder(rec)
    rec.PushNamespace("action_queue")
    s.ev.Settle(s, delta)           // 推进事件队列
    settleDiff, _ := mgr.Registry().BuildDiff(rec)
    s.ClearRecorder()
    // 推送增量更新（跳过纯进度更新）
}
```

`event.State.Settle` 消耗 `delta` 秒推进队列：
- 对队列头部事件计算 `timeCycles = int((delta + progress) / loop_time)`
- 受库存消耗限制 `maxAffordableByConsumables`
- 受 `target_cycles` 限制（若有）
- 一次性扣除消耗、发放奖励、累加经验、解锁图鉴
- 更新 `progress`，处理事件完成后的队列移除/切换

### 进度更新抑制

结算产生的 diff 若**仅包含** `"current"` scope 的事件队列进度更新（无物品、经验、属性等实质变更），则被判定为 `isProgressOnlyDiff`，**不会推送给客户端**。前端根据 `loop_time` 和上次收到的 `progress` 自行插值显示进度条。

### 批量刷盘

Tick Round 结束后，所有 `runTick` 返回 `dirty=true` 的 Session 被收集到 `results`：

```go
func (m *Manager) BatchFlush(ctx context.Context, database dbFlusher, results []TickResult) error {
    return database.InTx(ctx, func(q *dbgen.Queries) error {
        for _, r := range results {
            if err := flushSession(ctx, q, r.Sess); err != nil { return err }
        }
        return nil
    })
}
```

`flushSession` 按顺序刷入各个子系统：
1. `inventory.Flush` — UPSERT 脏库存槽位
2. `skill.Flush` — UPSERT 脏技能槽位
3. `bestiary.Flush` — UPSERT 脏解锁事件/发现物品
4. `event.Flush` — UPSERT 活跃队列，清理尾部位置
5. `equipment.Flush` — UPSERT/DELETE 装备槽位

## Session 状态机

```
StateActive  ──[disconnect]──>  StateGrace
    ▲                              │
    │                              │ [grace timer expires]
    │ [reconnect]                  ▼
    └────────────────────────  StateClosing ──[FlushAll+Close]──> StateClosed
```

| 状态 | 含义 | 行为 |
|------|------|------|
| `StateActive` | 连接正常 | Tick 推进，接受命令，推送 diff |
| `StateGrace` | 连接断开，宽限期中 | Tick 继续推进，Session 留在内存，可重连 |
| `StateClosing` | 宽限期到，正在关闭 | 执行 `FlushAll`，禁止新命令 |
| `StateClosed` | 已关闭 | Session 从 Manager 移除，资源释放 |

## 前端状态同步

### 连接时

后端发送 `state.full`，包含以下子系统的完整快照：

| 子系统 | 消息字段 | 说明 |
|--------|----------|------|
| 技能 | `skill_xp` | 所有技能等级和经验 |
| 库存 | `inventory` | 所有物品数量（>0 的条目） |
| 装备 | `equipment` | 槽位 → 物品映射 |
| 属性 | `attribute` | 所有属性最终值及修饰器列表 |
| 图鉴 | `bestiary` | 已解锁事件/物品/区域列表 |
| 事件队列 | `event_execution` | 队列条目及当前进度 |

### 运行时

后端发送 `state.diff`，包含本次变更的增量数据：

| 子系统 | diff 类型 | 触发时机 |
|--------|-----------|----------|
| `inventory` | `InventoryDiff` | 物品增减（含 equip/unequip 导致的库存变化） |
| `skill_xp` | `SkillXPDiff` | 经验获得/等级提升 |
| `attribute` | `AttributeDiff` | 修饰器变更后的最终值 |
| `bestiary` | `BestiaryDiff` | 新事件/物品/区域解锁 |
| `event_execution` | `EventExecutionDiff` | 事件完成循环次数 |
| `event_queue` | `EventQueueDiff` | 队列结构变化或进度更新 |
| `equipment` | `EquipmentDiff` | 装备/卸下操作 |

### 前端 WS 管理

前端在 `MainView.vue` `onMounted` 时注册消息监听器 `initWsListeners()`，在 `onBeforeUnmount` 时调用 `disposeWsListeners()` 和 `disconnect()` 优雅关闭连接（设置 `intentionalClose=true` 防止自动重连）。路由守卫在导航到需要认证的路由时调用 `connect()` 建立连接。
