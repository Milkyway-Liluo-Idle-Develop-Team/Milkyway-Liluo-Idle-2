# 技术说明

## 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 前端 | Vue 3 + Vite + Pinia | 三栏式布局，WebSocket 状态驱动 |
| 后端 | Go 1.25 + sqlc + Goose | SQLite (WAL mode)，单二进制部署 |
| 通信 | WebSocket (JSON/Proto 双模式) | 开发环境用 JSON，生产可用 Proto |
| 配置 | `actions.json` + `attributes.json` | 嵌入二进制，启动时加载校验 |

## 核心数据文件

| 文件/目录 | 说明 |
|-----------|------|
| `docs/actions.json` | 游戏核心数据配置，定义所有物品和事件的元数据 |
| `docs/actions_format.md` | `actions.json` 的格式说明文档 |
| `docs/attributes.json` | 属性定义及静态修饰器模板 |
| `docs/游戏文档/` | 各系统的详细设计文档 |

`actions.json` 和 `attributes.json` 均通过 `//go:embed` 嵌入二进制，启动时加载并校验。变更后需运行 `go run ./cmd/genregistry` 更新数字 ID 映射（`id_registry.json`、`attr_registry.json`）。

### 通信模型

核心游戏内容通过 WebSocket 与前端通信。

**服务端采用全局固定间隔 Tick**：
- `Manager.TickAll` 启动全局 `time.Ticker`（默认 50ms）。
- 每个 Tick Round 并行处理所有在线 Session（worker pool，按 CPU 核数分片）。
- `runTick` 内分 Phase 1（命令排空）和 Phase 2（事件结算）。
- 增量更新 `state.diff` 在 `runTick` 内部立即推送。
- Tick Round 结束后 `BatchFlush` 将所有脏 Session 在一个 SQLite 事务中批量刷盘。

**应用层心跳**：当前实现**没有应用层心跳**消息。服务端不依赖心跳判定连接存活，WebSocket `onclose` 回调触发 Grace Timer。底层 `wsx.Conn` 会按 `PingInterval`（默认 30s）发送 WebSocket ping 帧保活。

### WebSocket 生命周期

```
连接建立 → 认证 → 查找/创建 PlayerSession → 推送 state.full → 加入 Tick 循环
   → 在线期间（Tick 推进 + 命令处理 + 增量推送）
   → 断开 → DetachConn → StartGraceTimer(30s)
   → 宽限期内重连：复用 Session，停止 Grace，推送 state.full
   → 宽限期到：FlushAll → Remove → Close
```

单在线语义：同一 `userID` 不能有两个活跃 WebSocket 连接。新连接会踢掉旧连接并复用内存中的 `PlayerSession`。

### 属性系统

三层架构：
1. **属性定义层** — `attributes.json` 定义 id、默认值、min/max、方向、分组
2. **属性注册表** — 全局只读，启动时加载，维护属性依赖图和反向依赖图，检测循环依赖
3. **修饰器系统** — `ADD` → `MULTIPLY` → `OVERRIDE` 三阶段计算，支持 `ref_attr` 动态引用

计算采用**懒计算 + 脏标记**：只在被读取且 dirty 时重算，通过反向依赖图传播脏标记。

### 数据更新记录机制

各子系统实现 `record.SystemProvider` + `record.RecordBucket` 接口：

| 系统 | Provider/Bucket | 记录内容 | 合并规则 |
|------|-----------------|----------|----------|
| inventory | `inventory.Provider` / `Bucket` | 物品增减 | 同 `item.Item` 累加 qty |
| skill_xp | `skill.Provider` / `Bucket` | 经验变动 | 同 SkillID 累加 xp |
| attribute | `attribute.Provider` / `Bucket` | 脏属性标记 | 同 AttrID 去重 |
| bestiary | `bestiary.Provider` / `Bucket` | 解锁记录 | 同 Type+ID 去重 |
| event_execution | `event.execProvider` / `execBucket` | 执行次数 | 同 EventID+QueueID 累加 cycles |
| event_queue | `event.queueProvider` / `queueBucket` | 队列变更 | Scope 取最高优先级 |
| equipment | `equipment.Provider` / `Bucket` | 装备操作 | 同 slot 保留最后一次 |

## 前端

### 状态管理

Pinia `useGameStore` 管理游戏状态：
- `fetchStaticData()` — 加载 `actions.json` 静态配置（物品、事件、ID 注册表、等级曲线）
- `initWsListeners()` — 注册 `state.full` / `state.diff` / `error` 消息处理器
- `applyStateFull(full)` — 完整状态初始化（重置所有状态字段）
- `applyStateDiff(diff)` — 增量更新合并（inventory/skill_xp/attribute/bestiary/queue/equipment）

### 关键状态字段

```ts
state: {
  skills: {},           // skill_id -> { level, exp }
  inventory: [],        // { id, state, qty }[]
  equipment: {},        // slot -> item_id
  tools: {},            // tool_slot -> item_id
  event_counts: {},     // event_id -> 执行次数
  seen_items: [],       // 已发现物品字符串 ID（用于 crafting 可见性）
  unlocked_events: [],  // 已解锁事件字符串 ID
  attributes: {},       // attr_id -> final_value
  queue_items: [],      // { event_id, iterations, completed }[]
  queue_index: 0,
  queue_progress_seconds: 0,
}
```

### WS 连接管理

`web-client/src/lib/ws.ts` 管理全局 WebSocket：
- `connect()` — 建立连接（若已连接则直接返回）
- `disconnect()` — 优雅关闭（设置 `intentionalClose=true` 阻止自动重连）
- `send(type, payload)` / `sendAndWait(type, payload)` — 发送消息
- `onMessage(type, handler)` — 注册推送消息处理器
- `onStatusChange(handler)` — 监听连接状态变化

`MainView.vue` 在 `onMounted` 中调用 `initWsListeners()` 注册处理器，在 `onBeforeUnmount` 中调用 `disposeWsListeners()` + `disconnect()` 清理。

路由守卫在访问需要认证的路由时调用 `connect()`。
