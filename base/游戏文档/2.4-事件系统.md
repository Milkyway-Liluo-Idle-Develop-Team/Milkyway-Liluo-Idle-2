# 事件系统

事件（Event）是玩家与游戏交互的核心单元，表示玩家可以执行的各类行动和剧情节点。

## Event 数据格式

`actions.json` 中的每个 `Event` 对象定义如下：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 唯一标识符 |
| `type` | string | 是 | 事件类型：`"loop"` / `"instant"` / `"upgrade"` |
| `name` | string | 是 | 显示名称 |
| `description` | string | 是 | 描述文本 |
| `need_skill` | string | 是 | 关联技能 ID，`"none"` 表示无 |
| `requirements` | array | 是 | 执行前提条件 |
| `map` | string | 是 | 所属地图/场景 |
| `loop_time` | number | 否 | 循环周期（秒），`type == "loop"` 时必填 |
| `repeat_time` | number | 否 | 重复间隔（秒），用于可重复执行的升级 |
| `rewards` | array | 否 | 完成后的产出奖励 |

## 事件分类

### Loop 事件（循环事件）

最常见的生产类事件。玩家将其放入队列后，按 `loop_time` 周期性自动执行。

- 每次循环结算时：扣除消耗 → 发放奖励 → 累加经验。
- 多个 loop 事件可在不同队列槽位**并行**执行。
- `loop_time` 支持小数（如 1.5 秒），未凑满一次循环的进度会保留在 `progress` 字段中。

示例：`felling_oak_tree`（砍伐橡树），`loop_time: 2`，每 2 秒产出 1 个 `oak_logs` 和 20 点 `felling` 经验。

### Instant 事件（即时事件）

执行一次立即生效，不需要在队列中循环。通常用于一次性交互或特殊操作。

### Upgrade 事件（升级/解锁事件）

执行一次后**永久解锁**该事件 ID，将其加入玩家的 `unlocked_events` 集合。解锁后：

- 可能开放新的 loop 事件供玩家使用。
- 可能解锁新的物品配方。
- 推进主线或支线剧情。

部分 upgrade 事件可通过 `repeat_time` 支持可重复执行（多次升级同一建筑/能力）。

## 需求 (Requirements)

`requirements` 数组描述执行事件的前置条件。每个 Requirement：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | `"skill"` / `"item"` / `"fluid"` / `"event"` |
| `id` | string | 对应资源的 ID |
| `comparison_types` | string | 比较运算符，可选 |
| `value` | number | 比较/消耗数值，可选 |

### 结算规则

- **消耗型**：`type` 为 `item` 或 `fluid`，且 `comparison_types` 为 `null`。执行时从库存扣除 `value` 数量。
- **门槛型**：带有 `comparison_types`（如 `bigger_or_equal`）或不属于消耗型。仅做条件检查，不扣除资源。
- **事件依赖**：`type: "event"` 要求玩家已完成（解锁）指定事件。

### 事件可见性

前端展示的时候有一个可见性机制来限制玩家当前能够看到的任务。
一个任务需要满足以下要求才被展示：
1. requirements 内的 event、item、fluid已经在图鉴内解锁

## 奖励 (Rewards)

`rewards` 数组描述事件完成后的产出。Reward 有两种格式：

### 物品奖励

```json
{
  "id": "oak_logs",
  "num": 1
}
```

- `id`：物品 ID。
- `num`：产出数量（优先读取）。
- `value`：兼容字段，当 `num` 未设置时回退读取。

### 经验奖励

```json
{
  "type": "experience",
  "value": 20.0,
  "skill_id": "felling"
}
```

- `type`：固定为 `"experience"`。
- `value`：经验数值（支持小数）。
- `skill_id`：经验归属的技能。

## 事件队列

### 队列结构

事件队列为**串行队列**：玩家编排一个有序列表，系统按顺序从前往后执行，同一时间只处理队列头部的事件。未来可通过 `queue_id` 扩展为多队列并行机制。

队列数据存储在 `player_active_events` 表中：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | INTEGER | 玩家 ID |
| `queue_id` | INTEGER | 队列编号，`0` 为主队列，未来可扩展并行队列 |
| `event_id` | INTEGER | 事件数字 ID |
| `position` | INTEGER | 在队列中的顺序，从 `0` 开始递增 |
| `target_cycles` | INTEGER | 目标执行次数，`-1` 表示无限循环（默认），正整数表示执行指定次数后自动移除 |
| `progress` | REAL | 当前事件已积累的进度（秒） |

主键为 `(user_id, queue_id, position)`。同一 `queue_id` 下按 `position` 升序排列形成完整队列。

### 队列操作

- **编排**：玩家将已解锁事件按期望顺序加入队列，系统自动分配 `position`。
- **调整顺序**：支持拖拽或接口调用来修改事件的 `position`，重新排序队列。
- **移除**：从队列中移除某个事件，后续事件自动前移填补空缺。
- **清空**：一次性清空某条队列的所有事件。

### 结算规则（消耗时间模型）

队列本身**不计算时间差**，仅负责**消耗**外部传入的一段时间 `delta`，并产出相应内容。

结算流程：

1. 对每个 `queue_id`，按 `position` 升序取第一个事件作为**当前执行事件**（队列头部）。
2. 根据事件类型消耗 `delta` 并产出内容：

   **Loop 事件**（乐观批量结算）：

   ```
   actual_cycles = min(
       时间允许的次数( progress, delta, loop_time ),
       库存允许的次数( 各类消耗材料库存 / 单次消耗 ),
       target_cycles(若有限制)
   )

   一次性扣除 实际消耗 = 单次消耗 × actual_cycles
   一次性发放 实际产出 = 单次产出 × actual_cycles
   一次性累加 实际经验 = 单次经验 × actual_cycles
   更新 progress
   target_cycles 递减，归零后移出队列
   ```

   - **Instant 事件**：瞬间完成，从队列移除。
   - **Upgrade 事件**：瞬间完成，从队列移除并解锁对应事件。
3. **执行失败与队列重排**：
   - 在执行事件前，先检查其 requirements 是否全部满足（资源充足、技能达标、前置事件已解锁等）。
   - **若条件不满足**：从当前位置向后查找**第一个**满足执行条件的事件。若找到，则与当前事件**互换位置**（交换 `position`），然后以新头部事件继续结算；若遍历到队列末尾仍未找到，则**本次结算不执行任何操作**（时间 `delta` 不会被消耗，等待下次同步时再次尝试）。
   - **注意**：只有消耗型 requirement（如物品/流体不足）会导致交换；门槛型 requirement（如技能等级不足）不会触发交换，因为交换后该事件仍然无法执行。

4. 若事件执行后从队列移除（`target_cycles` 归零或为 Instant/Upgrade），下一个事件自动成为新的队列头部，继续用剩余 `delta` 参与结算。

**关键设计**：队列只关心"给定一段时间，能产出多少内容"，不维护独立的时间轴。时间差由调用方（如 sync 请求的离线时间计算）提供。

## 技术细节

- 事件使用稳定的数字 ID（`EventID`，`int64`）存储。
- 后端启动时构建索引：`eventsBySkill`、`eventsByMap`、`loopEvents`、`upgradeEvents`，查询为 O(1)。
- 事件校验在启动时完成：检测悬空引用（requirements/rewards 中指向不存在的物品/事件）、缺失必填字段等，校验失败会阻止服务器启动。
