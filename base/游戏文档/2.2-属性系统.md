# 属性系统

属性系统负责管理角色、装备、工具等实体的能力数值。其核心分为三层：**属性定义**、**属性注册**、**属性修饰**。

## 整体架构

```
┌─────────────────┐
│  属性定义层      │  定义有哪些属性、类型、默认值
├─────────────────┤
│  属性注册表      │  全局只读注册表，启动时加载
├─────────────────┤
│  修饰器系统      │  描述"如何改变属性值"
└─────────────────┘
```

---

## 1. 属性定义

每个属性由以下元数据描述：

| 字段 | 说明 |
|------|------|
| `id` | 字符串标识，如 `"physical_power"`，仅用于配置和前端展示 |
| `attr_id` | **数字 ID**（`AttributeID`，`int64`），内部存储和计算的唯一标识 |
| `name` | 显示名称 |
| `default_value` | 默认值（浮点数，支持小数）|
| `min_value` | **可选**。最小值限制，计算结果低于此值时截断 |
| `max_value` | **可选**。最大值限制，计算结果高于此值时截断 |
| `direction` | `positive`（越大越好）/ `negative`（越小越好），影响 UI 颜色展示 |
| `group` | 属性分组，如 `"combat"`、`"production"`、`"base"`，用于 UI 分组展示 |
| `desc` | 描述文本 |

运行时，每个玩家的每个属性实例额外维护：

| 字段 | 说明 |
|------|------|
| `cached_value` | 上次计算出的最终值 |
| `dirty` | 布尔标记，`true` 表示缓存已失效，下次读取需重算 |

---

## 2. 外部配置文件

属性定义和**静态修饰器**通过外部 JSON 文件（如 `attributes.json`）配置，与 `actions.json` 同等对待：

- 文件嵌入二进制（`//go:embed`），启动时加载。
- 变更后需运行 `go run ./cmd/genregistry` 更新数字 ID 映射。
- 字符串 ID 仅存在于配置层；代码与数据库全程使用数字 `AttributeID`。

### 文件格式

```json
{
  "attributes": [
    {
      "id": "initial_attack_power",
      "name": "初始攻击力",
      "default_value": 10,
      "min_value": 0,
      "direction": "positive",
      "group": "combat",
      "desc": "角色基础攻击力"
    },
    {
      "id": "extra_attack_power",
      "name": "额外攻击力",
      "default_value": 0,
      "direction": "positive",
      "group": "combat"
    },
    {
      "id": "final_attack_power",
      "name": "最终攻击力",
      "default_value": 0,
      "direction": "positive",
      "group": "combat"
    },
    {
      "id": "attack_interval",
      "name": "攻击间隔",
      "default_value": 2,
      "min_value": 0.1,
      "direction": "negative",
      "group": "combat",
      "desc": "数值越小攻击越快"
    }
  ],
  "modifiers": [
    {
      "attr_id": "final_attack_power",
      "op": "ADD",
      "ref_attr": "initial_attack_power"
    },
    {
      "attr_id": "final_attack_power",
      "op": "ADD",
      "ref_attr": "extra_attack_power"
    },
    {
      "attr_id": "attack_speed",
      "op": "MULTIPLY",
      "value": 0.2,
      "display": "PERCENT"
    }
  ]
}
```

| 段 | 说明 |
|--|------|
| `attributes` | 属性定义列表，每个属性对应注册表中的一行。 |
| `modifiers` | **静态修饰器模板**，游戏启动时为所有玩家统一挂载。典型用途：派生属性的公式表达（如 `final_attack = initial + extra`）。 |

### 数字 ID 注册

属性字符串 ID → 数字 ID 的映射由独立的 `attr_registry.json` 维护，规则与 `id_registry.json` 完全一致：

- 映射稳定不变，新增属性分配 `max+1`，已删除 ID 不回收。
- `attributes.json` 和 `actions.json` 中出现的属性字符串 ID 必须在 `attr_registry.json` 中存在，否则启动报错。

## 3. 属性注册表

游戏启动时从 `attributes.json` 加载属性定义到全局只读注册表，并挂载 `modifiers` 段中的静态修饰器。

- `actions.json` 中出现的属性字符串 ID 必须在注册表中存在，否则启动报错。
- 注册表额外维护**属性依赖图**：检测修饰器之间的循环引用，发现环时报错。
- 注册表同时维护**反向依赖图** `reverse_deps`，用于脏标记传播。

## 4. 修饰器系统

修饰器（Modifier）是改变属性值的最小单元。

### 修饰器结构

```
enum OpType {
    ADD,        // 加法修饰
    MULTIPLY,   // 乘法修饰
    OVERRIDE    // 覆盖修饰
}

enum DisplayMode {
    FIXED,      // 固定数值，如 "+15"
    PERCENT,    // 百分比，如 "+20%"
    PER_SECOND  // 每秒，如 "+5/秒"
}

Modifier {
    attr_id     AttributeID  // 目标属性数字 ID
    op          OpType       // 操作类型枚举
    value       number?      // 固定数值（与 ref_attr 二选一）
    ref_attr    AttributeID? // 引用的属性数字 ID（与 value 二选一）
    display     DisplayMode  // 前端展示方式，默认 FIXED
    source      string       // 来源标识
    priority    int          // 计算优先级
}
```

**关键设计**：修饰器的 `value` 可以是**固定数值**，也可以是**对其他属性的引用**（`ref_attr`）。通过 `ref_attr`，一个属性的修饰器可以动态依赖另一个属性的当前值。

### 操作类型

| 枚举值 | 说明 | 计算阶段 |
|--------|------|----------|
| `ADD` | 加法修饰，累加到当前值 | 第一阶段 |
| `MULTIPLY` | 乘法修饰，效果为 `×(1 + value)` | 第二阶段 |
| `OVERRIDE` | 覆盖修饰，直接替换最终值 | 第三阶段 |

### 展示方式 (display)

| 枚举值 | 说明 | 前端展示示例 |
|--------|------|--------------|
| `FIXED` | 固定数值 | `+15`、`-5` |
| `PERCENT` | 百分比 | `+20%`、`-10%` |
| `PER_SECOND` | 每秒 | `+5/秒` |

`display` 仅影响前端展示文案，不参与实际数值计算。

### 修饰器来源

| 来源 | 场景 |
|------|------|
| `equipment` | `equipment_basic_data` / `equipment_upgrade_data` |
| `tool` | `tool_basic_data` |
| `skill` | 技能等级加成 |
| `buff` | 临时状态 |
| `base` | 角色初始值 |

### 修饰器收集

```
modifiers = []
modifiers += 角色基础修饰器
modifiers += 已穿戴装备解析出的修饰器
modifiers += 已装备工具解析出的修饰器
modifiers += 当前生效 Buff 的修饰器
modifiers += 技能等级带来的修饰器
```

### 修饰器生命周期管理

每个玩家的动态修饰器在内存中按 `source` 分组存储。外部系统（装备、技能、Buff 等）通过以下接口维护修饰器，属性系统负责在变更后自动标记脏缓存：

```
interface ModifierLifecycle {
    AddModifiers(source string, mods []Modifier)     // 添加一组修饰器
    RemoveModifiers(source string)                    // 移除某来源的全部修饰器
    UpdateModifiers(source string, mods []Modifier)   // 原子替换某来源的修饰器
}
```

**实现细节**：`source` 作为分组键，同一来源的修饰器覆盖写，不叠加。例如 `"equipment:sword_001"` 代表某件具体装备的修饰器组。

#### 各来源维护方式

| 来源 | 触发时机 | 操作 |
|------|----------|------|
| `equipment` | 穿戴装备 | `AddModifiers("equipment:{item_id}", mods)` |
| `equipment` | 卸下装备 | `RemoveModifiers("equipment:{item_id}")` |
| `tool` | 装备/卸下工具 | 同 equipment |
| `skill` | 技能等级变化 | `UpdateModifiers("skill:{skill_id}", newMods)` |
| `buff` | 获得 Buff | `AddModifiers("buff:{buff_id}", mods)` |
| `buff` | Buff 过期/驱散 | `RemoveModifiers("buff:{buff_id}")` |
| `base` | 角色创建 | `AddModifiers("base", baseMods)`，通常不变 |

#### 变更后的脏标记

任何 Add / Remove / Update 操作完成后，属性系统执行：

```
function on_modifiers_changed(changed_attr_ids):
    for attr_id in changed_attr_ids:
        mark_dirty(attr_id)
```

`changed_attr_ids` 是被变更的修饰器所指向的**目标属性 ID** 集合。属性系统内部根据反向依赖图自动把受影响的上游属性一并标记为脏。

#### WebSocket 连接级属性实例

属性实例随 **WebSocket 连接生命周期** 创建和销毁，不维护全局在线池。

**连接建立时**：

```
1. WebSocket 握手成功，为该连接创建 PlayerSession
2. 从数据库读取玩家状态（穿戴装备、技能等级、生效 Buff）
3. 在 PlayerSession 内新建属性实例，挂载静态修饰器
4. 遍历当前穿戴装备 → AddModifiers("equipment:...", mods)
5. 遍历当前装备工具 → AddModifiers("tool:...", mods)
6. 遍历玩家技能等级 → UpdateModifiers("skill:...", mods)
7. 遍历当前生效 Buff → AddModifiers("buff:...", mods)
8. 全部标记为 dirty（首次读取时懒计算）
```

**连接期间**：

属性实例常驻于 PlayerSession 内存中。写操作（穿脱装备、技能升级、Buff 变化）直接操作该实例，**不重建**：

| 业务场景 | 操作 |
|----------|------|
| 玩家穿戴装备 | `AddModifiers("equipment:{id}", mods)`，然后写数据库 |
| 玩家卸下装备 | `RemoveModifiers("equipment:{id}")`，然后写数据库 |
| 技能升级 | `UpdateModifiers("skill:{id}", newMods)`，然后写数据库 |
| 获得 Buff | `AddModifiers("buff:{id}", mods)`，然后写数据库 |
| Buff 到期 | `RemoveModifiers("buff:{id}")`，然后写数据库 |

同步、结算、战斗等读场景直接读取实例内的 `cached_value`，不查库、不重建。

**连接断开时**：

PlayerSession 释放，属性实例连同修饰器集合一起被 GC 回收。下次连接重新走「连接建立」流程。

## 6. 属性计算流程

### 计算伪代码

```
function get_final(attr_id):
    attr = instance[attr_id]

    // 缓存命中且未脏，直接返回
    if !attr.dirty:
        return attr.cached_value

    // 开始计算
    value = registry[attr_id].default_value
    mods = filter(modifiers, m.attr_id == attr_id)
    sort(mods, by priority asc)

    // ADD 修饰器
    for m in filter(mods, op == ADD):
        if m.ref_attr != nil:
            value += get_final(m.ref_attr)
        else:
            value += m.value

    // MULTIPLY 修饰器：效果为 ×(1 + value)
    for m in filter(mods, op == MULTIPLY):
        if m.ref_attr != nil:
            value *= (1 + get_final(m.ref_attr))
        else:
            value *= (1 + m.value)

    // OVERRIDE 修饰器
    ov = filter(mods, op == OVERRIDE)
    if ov not empty:
        value = ov[0].value

    // 截断到 [min_value, max_value] 范围（若配置了）
    if registry[attr_id].min_value != nil:
        value = max(value, registry[attr_id].min_value)
    if registry[attr_id].max_value != nil:
        value = min(value, registry[attr_id].max_value)

    // 写缓存，清脏标记
    attr.cached_value = value
    attr.dirty = false
    return value
```

### 脏标记传播

当某属性的修饰器发生变更（装备更换、Buff 增减、技能升级等）：

```
function mark_dirty(attr_id):
    instance[attr_id].dirty = true
    // 将所有引用了该属性的其他属性也标记为脏
    for ref_id in reverse_deps[attr_id]:
        mark_dirty(ref_id)
```

注册表在启动时构建**反向依赖图** `reverse_deps`：若属性 `5` 的修饰器引用了属性 `3`，则 `reverse_deps[3]` 包含 `5`。这样属性 `3` 变化时，属性 `5` 自动连带变脏。

### 临时计算上下文

某些场景（如事件结算、战斗模拟）需要在**单次计算**内临时覆盖默认值或注入临时修饰器，且不影响持久状态。

支持通过**计算上下文**（Context）传入临时参数：

```
Context {
    temp_base    map[AttributeID]number    // 临时覆盖默认值
    temp_mods    []Modifier                // 本次计算额外生效的临时修饰器
}

function get_final_with_context(attr_id, ctx, visited):
    attr = instance[attr_id]

    // 若传入了临时默认值，优先使用；否则用注册表默认值
    base = ctx.temp_base[attr_id] ?? registry[attr_id].default_value

    // 持久修饰器 + 临时修饰器合并
    mods = filter(modifiers, m.attr_id == attr_id)
    mods += filter(ctx.temp_mods, m.attr_id == attr_id)
    sort(mods, by priority asc)

    // 后续计算逻辑与 get_final 相同...
```

**使用规则**：
- 临时上下文**不参与缓存**，每次调用独立计算。
- 临时上下文**不触发脏标记**，不影响其他属性或后续读取。
- 临时修饰器可被 `ref_attr` 正常引用（递归计算时会一起参与）。

**典型场景**：
- 事件结算时，某 Buff 临时提供一次性的产出加成 → 通过 `temp_mods` 注入。
- 战斗模拟时，临时假设角色等级提升后的属性 → 通过 `temp_base` 覆盖默认值。
- 预览强化结果时，计算装备升级后的属性变化 → 通过 `temp_mods` 注入升级修饰器。

### 计算原则

- **递归求值**：`ref_attr` 触发的引用会递归调用 `get_final`，先算出被引用属性的最终值，再参与当前属性的计算。
- **循环依赖检测**：`get_final` 内部维护 `visited` 集合，若发现环则报错。
- **懒计算**：属性只在被读取且 `dirty == true` 时才重算，日常结算直接读 `cached_value`。

## 7. 复杂示例

实现：最终攻击力 = （初始攻击力 + 额外攻击力）× 攻击力乘区1 × 攻击力乘区2

**注册表定义**：

```go
// 假设注册表分配的数字 ID：
// initial_attack_power = 1, extra_attack_power = 2,
// attack_zone1 = 3, attack_zone2 = 4, final_attack_power = 5

registry.Register({ID: "initial_attack_power",  AttrID: 1, DefaultValue: 10})
registry.Register({ID: "extra_attack_power",    AttrID: 2, DefaultValue: 0})
registry.Register({ID: "attack_zone1",          AttrID: 3, DefaultValue: 0})
registry.Register({ID: "attack_zone2",          AttrID: 4, DefaultValue: 0})
registry.Register({ID: "final_attack_power",    AttrID: 5, DefaultValue: 0})
```

**挂载在 `final_attack_power`（ID=5）上的修饰器**：

```
Modifier{attr_id: 5, op: "add",      ref_attr: 1}
Modifier{attr_id: 5, op: "add",      ref_attr: 2}
Modifier{attr_id: 5, op: "multiply", ref_attr: 3}
Modifier{attr_id: 5, op: "multiply", ref_attr: 4}
```

**其他属性的修饰器（示例）**：

```
// extra_attack_power (ID=2) 被装备加了 15
Modifier{attr_id: 2, op: "add", value: 15, source: "equipment:xxx"}

// attack_zone1 (ID=3) 被技能加了 0.2（默认值 0 + 0.2 = 0.2，乘区效果 1 + 0.2 = 1.2）
Modifier{attr_id: 3, op: "add", value: 0.2, source: "skill:xxx"}

// attack_zone2 (ID=4) 被 Buff 加了 0.1（默认值 0 + 0.1 = 0.1，乘区效果 1 + 0.1 = 1.1）
Modifier{attr_id: 4, op: "add", value: 0.1, source: "buff:xxx"}
```

**计算过程**：

```
get_final(5):
  add ref=1     → get_final(1) = 10
  add ref=2     → get_final(2) = 0 + 15 = 15
  multiply ref=3 → get_final(3) = 0 + 0.2 = 0.2  → effect = 1 + 0.2 = 1.2
  multiply ref=4 → get_final(4) = 0 + 0.1 = 0.1  → effect = 1 + 0.1 = 1.1

  result = (0 + 10 + 15) * 1.2 * 1.1 = 33
```

## 8. 数据绑定（actions.json → 修饰器）

`actions.json` 中的键值对解析为修饰器时，属性字符串 ID 必须先查 `attr_registry.json` 转换为数字 `AttributeID`：

```json
// actions.json
"equipment_basic_data": {
  "extra_attack_power": 10,
  "attack_zone1": 0.2
}
```

解析后生成：

```
Modifier{attr_id: 2, op: "add", value: 10}   // extra_attack_power = 2
Modifier{attr_id: 3, op: "add", value: 0.2}  // attack_zone1 = 3
```

如果 `actions.json` 中出现未在 `attributes.json` 中注册的属性 ID，启动校验时报错。

> 注：`attributes.json` 的 `modifiers` 段与 `actions.json` 解析出的修饰器在运行时合并到同一个修饰器列表中，无优先级区分。

## 9. 缓存与性能

- **属性级缓存**：每个属性的 `cached_value` 持久保存，不是单次结算临时缓存。
- **脏标记驱动重算**：修饰器变更时调用 `mark_dirty(attr_id)`，通过反向依赖图把该属性及所有引用它的属性全部标记为脏。读取时若 `dirty == true` 才执行 `get_final` 重算。

## 10. 持久化边界

属性系统**设计为不直接依赖数据库**。注册表、修饰器模板、计算逻辑全部在内存中运行。属性系统所需的一切输入，由调用方通过**玩家状态快照**提供。

### 哪些数据属于属性系统（纯内存，不入库）

| 数据 | 存储位置 | 说明 |
|------|----------|------|
| 属性注册表 | 内存 | 来自 `attributes.json`，启动时加载 |
| 静态修饰器模板 | 内存 | 来自 `attributes.json` 的 `modifiers` 段 |
| 装备/工具修饰器模板 | 内存 | 来自 `actions.json`，启动时解析 |
| 属性依赖图 / 反向依赖图 | 内存 | 启动时构建 |
| 玩家属性缓存 (`cached_value` + `dirty`) | 内存 | 常驻于 WebSocket 连接的 PlayerSession 中 |

### 哪些数据需要入库（数据库持久化）

| 数据 | 数据库表 | 说明 |
|------|----------|------|
| 玩家穿戴的装备 | `player_inventory` | 属性系统只关心哪些装备被穿戴，不关心库存本身 |
| 玩家技能等级 | `player_skills` | 技能等级决定技能类修饰器的数值 |
| 玩家已解锁事件 | `player_unlocked_events` | 影响某些事件相关的门槛或 Buff |
| 临时 Buff 状态 | 需新增表（如 `player_buffs`）| 若有带持续时间的 Buff，需记录到期时间 |

**关键原则**：数据库只存"状态事实"（穿了什么、等级多少）。属性系统负责把事实翻译成修饰器并计算最终值。玩家属性实例常驻于 WebSocket 连接（PlayerSession）内存中，连接断开后销毁，下次连接重新从状态事实构建。

## 11. 与事件结算的交互

```
final_power = session.Attr.GetFinal("final_attack_power")
damage = base_damage * final_power
```

事件系统通过玩家会话（PlayerSession）中的属性实例读取最终值，不感知修饰器内部结构。
