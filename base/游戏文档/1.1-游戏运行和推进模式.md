# 游戏运行和推进模式

## WebSocket 生命周期

```
┌─────────────┐     握手+认证      ┌─────────────┐
│   客户端     │ ───────────────> │   服务端     │
│  (断开状态)  │                   │  (等待连接)  │
└─────────────┘                   └─────────────┘
                                        │
                                        ▼

                              │  1. 加载玩家数据    │
                              │     (DB查询)        │

                              └───────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────┐
│                      在线期间                        │
│  ┌──────────┐    heartbeat(5s)    ┌──────────┐      │
│  │  客户端   │ ──────────────────> │  服务端   │     │
│  │          │ <────────────────── │          │      │
│  └──────────┘       ack(ts)       └──────────┘      │
│                                                     │
│         服务端推进 ──> 有变更 ──> 推送增量更新         │
└─────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐

   ┌─────────────┐                 ┌─────────────┐

               │  记录最后推进时间    │

               │  2. 执行离线结算    │

               │  4. 启动推进循环    │
               └───────────────────┘
```

### 连接建立

1. **握手与认证**：客户端发起 WebSocket 连接，携带 token 进行身份验证。
2. **加载玩家数据**：从数据库读取玩家完整状态（库存、技能、事件队列、Buff 等）。
3. **构建 PlayerSession**：创建会话对象，挂载属性实例（挂载修饰器、标记 dirty）。
4. **推送完整状态**：向客户端发送当前完整游戏快照（事件队列、库存、属性值等）。
5. **启动推进循环**：为该连接启动定时推进协程，开始主动结算。

### 在线期间

- **服务端定时推进**：按固定间隔（如 1 秒）推进事件队列，有产出或状态变更时主动推送增量更新。
- **客户端心跳**：每 5~10 秒发送 `heartbeat`，服务端回复服务器时间戳 ack，仅用于保活和校时。
- **客户端请求**：玩家主动操作（穿脱装备、调整队列、使用物品等）通过 WebSocket 消息发送，服务端处理后直接推送增量更新。

### 连接断开

- **正常断开**：客户端主动调用 `close()`，服务端收到断开事件，优雅释放资源。
- **异常断开**：心跳超时（如连续 3 次未收到 heartbeat）或网络中断，服务端判定连接失效。
- **资源释放**：停止推进循环，记录最后推进时间戳，释放 PlayerSession 及属性实例。

### 重连与离线结算

玩家断开期间，事件队列在服务端**暂停推进**。下次建立连接时：

1. 读取上次推进时间戳，计算离线时长。
2. 执行离线结算：将离线时长作为 `delta` 一次性推进事件队列。
3. 推送更新后的完整状态。
4. 重新启动在线推进循环。

## 推进机制

游戏采用**服务端定时驱动推进**模式。服务器为每个在线玩家维护独立的游戏循环，按固定间隔主动推进事件队列、结算产出。客户端仅通过心跳维持连接和同步状态显示，**不作为触发结算的信源**。

### 在线期间

玩家 WebSocket 连接建立后，服务端为该连接启动一个**玩家推进循环**：

```
interval = 1.0s  // 推进间隔，可按需调整

while 连接存活:
    sleep(interval)
    delta = interval
    推进事件队列(delta)
    结算产出并应用到玩家数据
    若本次推进有实质性变更（产出、队列切换等）:
        推送增量更新给客户端
```

### 推进子进程

每个在线玩家的推进逻辑运行在一个**独立的推进协程**中，与 WebSocket 连接管理分离。协程按固定间隔被唤醒，计算应推进的时间，逐个驱动各游戏系统结算。

```
计算 elapsed = 当前时间 - 上次推进时间
    │
    ▼
目标时间 final_time = 上次推进时间 + elapsed
    │
    ▼
逐个推进各系统（以事件队列为例）:
    │
    ├── PushNamespace("action_queue")
    │
    ├── 推进事件队列(elapsed)
    │      • 事件系统写入 EventExecutionRecord
    │      • 物品系统写入 InventoryChangeRecord
    │      • 经验系统写入 SkillXpRecord
    │      • 属性系统写入 ModifierChangeRecord
    │      • 图鉴系统写入 BestiaryUnlockRecord
    │
    ├── PopNamespace()
    │
    └── 其他可推进系统（如 战斗 等）重复上述命名空间过程
    │
    ▼
处理更新记录
    │
    ├── 同类合并（同一 ItemID 累加、同一 AttrID 去重...）
    │
    ├── 命名空间聚合（汇总各系统的所有变更）
    │
    ├── 构建差量包（按各系统序列化格式组装 JSON）
    │
    ▼
有实质性变更？
    │
    ├──是──> 推送增量更新给客户端
    │
    └──否──> 不推送
    │
    ▼
上次推进时间 = final_time
```

**框架说明**：

1. **时间计算**：每次唤醒时计算 `elapsed = 当前时间 - 上次推进时间`。使用目标时间 `final_time` 而非直接累加 `interval`，避免定时器漂移累积误差。
2. **逐个推进**：玩家可能拥有多个可推进的子系统（事件队列、Buff 倒计时、属性恢复等）。每个子系统的推进独立进行，各自管理命名空间。
3. **命名空间隔离**：每个子系统在推进前后 `Push/PopNamespace`，确保记录不跨系统混杂。同一轮推进可能产生多个命名空间，最终聚合时合并处理。
4. **记录处理**：
   - **同类合并**：如多次 `InventoryChangeRecord` 按 `ItemID` + `ItemState` 累加。
   - **命名空间聚合**：将本轮推进的所有命名空间记录汇总。
   - **差量包构建**：按各系统定义的序列化格式组装为 JSON。
5. **增量推送**：若聚合后的记录非空，通过 WebSocket 推送给客户端；若无实质性变更，本轮不推送。
6. **时间戳更新**：将 `上次推进时间` 设为 `final_time`，无论是否有变更都刷新，用于后续离线结算。

- 推进循环在服务端内部运行，不需要客户端请求触发。
- 客户端定时发送 `heartbeat`（如每 5 秒），服务端仅回复极简 ack（如服务器时间戳）。
- 心跳提供**保活 + 校时**。

### 前端心跳 (`heartbeat`)

```
客户端          服务端
  |  ──heartbeat──>  |
  |                  |  
  |  <────ack──────  |  返回服务器时间戳
```

- 心跳间隔：建议 5~10 秒，由客户端控制。
- 若心跳超时未收到，服务端仍继续推进，积累一定超时心跳后断连。

### 离线处理

玩家断开 WebSocket 后，推进循环停止。下次建立连接时：

```
1. 读取上次推进时间戳
2. 计算离线时长 = 当前时间 - 上次推进时间
3. 执行一次离线结算（等同于把离线期间当作一个大的 delta 推进事件队列）
4. 应用全部离线产出到玩家数据
5. 启动在线推进循环
```

离线期间的事件产出在重连时一次性结算，无需客户端等待。
