# 物品系统

物品系统涵盖游戏中所有可获取、可使用的实体定义，以及玩家库存的管理机制。

## Item 数据格式

`actions.json` 中的每个 `Item` 对象定义如下：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 唯一标识符，如 `"oak_logs"` |
| `name` | string | 是 | 显示名称 |
| `tool` | bool | 是 | 是否为工具 |
| `equipment` | bool | 是 | 是否为装备 |
| `upgradable` | bool | 是 | 是否可升级/强化 |
| `classification` | string | 是 | 分类标签 |
| `tool_details` | object | 否 | `tool == true` 时必填 |
| `equipment_details` | object | 否 | `equipment == true` 时必填 |
| `upgrade_details` | object | 否 | `upgradable == true` 时必填 |
| `extra` | array | 否 | 额外属性，如 `[{"id":"heat","value":8}]` |

### 分类 (classification)

当前使用的分类包括：

| 分类值 | 说明 |
|--------|------|
| `resources` | 原材料（木头、石头等） |
| `equipment` | 可穿戴装备 |
| `fuel` | 燃料（如煤炭） |
| `ores` | 矿石 |
| `important` | 关键道具 |
| `animal_materials` | 动物材料 |
| `fluid` | 流体（特殊物品，如热量、铁流体、石流体） |

### 流体作为特殊物品

流体（如 `heat`、`iron_fluid`、`stone_fluid`）本质上是物品的一种特殊形式，使用相同的 `ItemID` 和库存系统存储。它们在 `actions.json` 中与普通物品并列定义，`classification` 为 `"fluid"`。事件中的 `type: "fluid"` 需求在结算时同样扣除库存中的对应物品数量。

### 工具详情 (tool_details)

| 字段 | 说明 |
|------|------|
| `tool_position_requirement` | 工具生效位置要求 |
| `tool_type` | 工具类型，如 `"axe"`、`"pickaxe"` |
| `tool_basic_data` | 对产出的加成系数 |
| `tool_upgrade_data` | 升级后的加成数据 |
| `requirements` | 装备该工具的前置条件 |

### 装备详情 (equipment_details)

| 字段 | 说明 |
|------|------|
| `type` | 装备类型：`"weapon"`、`"wear"`、`"relics"` 等 |
| `element` | 元素属性：`"physical"`、`"magic"` 等 |
| `battle_skills` | 装备附带的战斗技能列表 |
| `equipment_position_requirements` | 穿戴位置要求 |
| `equipment_basic_data` | 基础属性数值 |
| `equipment_upgrade_data` | 升级后属性数值 |
| `requirements` | 穿戴前置条件 |

### 升级详情 (upgrade_details)

| 字段 | 说明 |
|------|------|
| `max_upgrade` | 最大升级等级 |
| `enhance_slot` | 强化槽位数 |
| `forge_slot` | 锻造槽位数 |
| `upgrade_curve` | 每阶段的推荐等级、成功率、属性倍率 |

## Inventory 系统

玩家库存由 `player_inventory` 表管理：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | INTEGER | 玩家 ID |
| `item_id` | INTEGER | 物品数字 ID |
| `item_state` | INTEGER | 32 位变体描述符（如强化等级） |
| `quantity` | INTEGER | 整数数量 |
| `fraction` | REAL | 小数累计部分（0 ≤ fraction < 1） |
| `updated_at` | DATETIME | 最后更新时间 |

主键为 `(user_id, item_id, item_state)`。

### 小数累计机制

某些加成（如 Buff、工具效果）可能导致单次产出为**非整数**（例如 1.3 个石头）。数据库中：`quantity` 存储整数部分，`fraction` 存储小数部分。

**结算规则**：
1. 计算原始产出 `delta`（可能含小数）。
2. `total = quantity + fraction + delta`。
3. `quantity = floor(total)`，`fraction = total - quantity`。
4. 保存更新后的 `quantity` 和 `fraction`。

这样玩家看到的数量始终是整数，但后台精确保留了小数累计，避免长期舍入误差。

### 状态位 (item_state)

`item_state` 用于区分同一 `item_id` 的不同变体：

- `0`：默认状态（未强化、未锻造的基础物品）。
- 非零值：可编码强化等级、锻造词缀索引等信息。具体编码规则由业务层定义。

## 更新记录

物品系统在库存发生变动时向当前命名空间写入记录。

### 库存变化记录

```go
type InventoryChangeRecord struct {
    ItemID        ItemID
    ItemState     int32
    QuantityDelta int
    FractionDelta float64
}
```

**产生时机**：事件结算发放/扣除物品、制造/合成、物品使用时写入。

### 合并规则

同一 `ItemID` + `ItemState` 在**同一命名空间**内可累加：

```
merged.QuantityDelta = sum(records[i].QuantityDelta)
merged.FractionDelta = sum(records[i].FractionDelta)
```

合并后若 `FractionDelta` 跨越整数边界（如累计到 1.0 以上或 -1.0 以下），由业务层在写数据库前统一处理进位，更新记录中保留原始小数变化即可。

### 序列化示例

```json
{
  "inventory_changes": [
    {"item_id": 3, "item_state": 0, "quantity_delta": 5},
    {"item_id": 1, "item_state": 0, "quantity_delta": -5}
  ]
}
```

## 技术细节

- 后端启动时通过 `gameconfig.Load()` 加载并校验 `actions.json`，所有物品数据常驻内存，访问为 O(1)。
- 物品使用稳定的数字 ID（`ItemID`，`int64`）存储在数据库中，字符串 ID 仅用于前端展示和配置编辑。
- `actions.json` 变更后需运行 `go run ./cmd/genregistry` 重新生成数字 ID 映射。
