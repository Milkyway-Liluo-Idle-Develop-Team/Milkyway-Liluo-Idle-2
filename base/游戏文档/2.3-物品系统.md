# 物品系统

物品系统涵盖物品的身份模型、定义、库存管理及装备机制。

## 物品身份模型

一个物品实例由 **`(item_id, item_state)`** 联合标识。两者必须同时存在：

- `item_id`：物品模板的数字 ID（`item.ID`，`int32`），对应 `actions.json` 中的物品定义
- `item_state`：32 位变体描述符（`item.State`，`int32`），编码强化等级、锻造词缀等变体信息

**`item_state = 0` 是默认态，不是"缺省"。** 从逻辑上不存在"只有 item_id 没有 item_state"的物品——表述时允许省略 State=0 的写法，但代码中始终以 `item.Item{ID: id, State: state}` 的完整形式传递。

## ItemDef（物品定义）

`actions.json` 中的每个 Item 对象在运行时被解析为 `item.ItemDef`——一个只读的物品定义：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 唯一标识符，如 `"oak_logs"` |
| `name` | string | 是 | 显示名称 |
| `tool` | bool | 是 | 是否为工具 |
| `equipment` | bool | 是 | 是否为装备 |
| `upgradable` | bool | 是 | 是否可升级/强化 |
| `classification` | string | 是 | 分类标签 |
| `tool_details` | object | 否 | `tool == true` 时必填 |
| `equipment_details` | object | 否 | `equipment == true` 时必填 |
| `upgrade_details` | object | 否 | `upgradable == true` 时必填 |
| `extra` | array | 否 | 额外属性，如 `[{"id":"heat","value":8}]` |

`ItemDef` 上的 `Modifiers(it Item, attrReg)` 方法接受完整的 `Item` 身份参数。当前 `it.State` 暂时忽略，将来通过 `upgrade_curve` 和 `ability_multiplier` 计算修饰器值。

### 分类 (classification)

当前使用的分类包括：

| 分类值 | 说明 |
|--------|------|
| `resources` | 原材料（木头、石头等） |
| `equipment` | 可穿戴装备 |
| `fuel` | 燃料（如煤炭） |
| `ores` | 矿石 |
| `important` | 关键道具 |
| `animal_materials` | 动物材料 |
| `fluid` | 流体（特殊物品，如热量、铁流体、石流体） |

### 流体作为特殊物品

流体（如 `heat`、`iron_fluid`、`stone_fluid`）本质上是物品的一种特殊形式，使用相同的 `item.Item` 身份和库存系统存储。它们在 `actions.json` 中与普通物品并列定义，`classification` 为 `"fluid"`。事件中的 `type: "fluid"` 需求在结算时同样扣除库存中的对应物品数量。

### 工具详情 (tool_details)

| 字段 | 说明 |
|------|------|
| `tool_position_requirement` | 工具生效位置要求 |
| `tool_type` | 工具类型，如 `"axe"`、`"pickaxe"` |
| `tool_basic_data` | 对产出的加成系数 |
| `tool_upgrade_data` | 升级后的加成数据 |
| `requirements` | 装备该工具的前置条件 |

### 装备详情 (equipment_details)

| 字段 | 说明 |
|------|------|
| `type` | 装备类型：`"weapon"`、`"wear"`、`"relics"` 等 |
| `element` | 元素属性：`"physical"`、`"magic"` 等 |
| `battle_skills` | 装备附带的战斗技能列表 |
| `equipment_position_requirements` | 穿戴位置要求 |
| `equipment_basic_data` | 基础属性数值 |
| `equipment_upgrade_data` | 升级后属性数值 |
| `requirements` | 穿戴前置条件 |

### 升级详情 (upgrade_details)

| 字段 | 说明 |
|------|------|
| `max_upgrade` | 最大升级等级 |
| `enhance_slot` | 强化槽位数 |
| `forge_slot` | 锻造槽位数 |
| `upgrade_curve` | 每阶段的推荐等级、成功率、属性倍率 |

## Inventory 系统

玩家库存由 `player_inventory` 表管理：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | INTEGER | 玩家 ID |
| `item_id` | INTEGER | 物品数字 ID |
| `item_state` | INTEGER | 32 位变体描述符 |
| `quantity` | REAL | 数量（支持小数，展示时 floor） |
| `updated_at` | DATETIME | 最后更新时间 |

主键为 `(user_id, item_id, item_state)`。

### 产出精度

结算时产出加成可能产生非整数（如 1.3 个石头）。**`quantity` 使用 REAL 存储**，保留小数部分跨结算周期累积。玩家看到的数量为 `floor(quantity)`。

```
raw = base × actualCycles × (1 + multiplier)
quantity = current + raw    -- REAL，如实累加
display = floor(quantity)    -- 整数展示
```

理由：结算可能只推进单轮（如推送时机恰好在一次循环完成后），此时截断损失无法被批量推进抵消。REAL 存储确保精度不丢失。

## 装备机制

装备是放在特定槽位的物品。将物品放入装备槽时，从 `ItemDef` 解析修饰器并挂载到属性系统：

```
Equip(it Item, slot string):
    def = gameconfig.GetItemDefByID(it.ID)
    mods = def.Modifiers(it, attrReg)
    attr.AddModifiers("equipment:" + def.StringID(), mods)

Unequip(slot):
    attr.RemoveModifiers("equipment:" + def.StringID())
```

## 更新记录

物品系统在库存发生变动时向当前命名空间写入记录。

### 库存变化记录

```go
type InventoryChangeRecord struct {
    Item          item.Item
    QuantityDelta float64
}
```

**产生时机**：事件结算发放/扣除物品、制造/合成、物品使用时写入。

**说明**：`QuantityDelta` 为 float64，如实记录变化量（含小数部分）。前端展示时取 `floor`。

### 合并规则

同一 `item.Item`（`ID` + `State`）在**同一命名空间**内累加 `QuantityDelta`。

### 序列化示例

```json
{
  "inventory_changes": [
    {"item_id": 3, "item_state": 0, "quantity_delta": 5},
    {"item_id": 1, "item_state": 0, "quantity_delta": -5},
    {"item_id": 19, "item_state": 10, "quantity_delta": 1}
  ]
}
```

## 技术细节

- 后端启动时通过 `gameconfig.Load()` 加载并校验 `actions.json`，所有物品数据常驻内存，访问为 O(1)
- 物品使用稳定的数字 ID（`item.ID`，`int32`）存储在数据库中，字符串 ID 仅用于前端展示和配置编辑
- `actions.json` 变更后需运行 `go run ./cmd/genregistry` 重新生成数字 ID 映射
- 游戏配置层的 `Item` JSON 结构不对外暴露；调用方使用 `item.ItemDef`（运行时定义）和 `item.Item`（实例身份）
